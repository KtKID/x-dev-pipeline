# x-qdev — 轻量级快速开发

适用于小功能、小修改、bug 修复等场景。省去需求分析和详细计划阶段，直接进入开发。

## 工作流程

```
用户描述任务 → 创建任务目录 → 编写 README.md（说明+清单）→ 逐项开发 → 更新状态 → 代码审查 → 完成
```

## 第一步：理解任务

收到用户的开发请求后：

1. 快速理解用户要做什么
2. 如果描述模糊，简短地问 1-2 个关键问题（不要像 x-req 那样做深度需求分析）
3. 评估涉及的文件和改动范围

如果改动明显很大（预计超过 5 个任务项、涉及架构变更），建议用户改用 x-req → x-plan → x-dev 流程。

## 第二步：创建任务记录

在 `dev-pipeline/tasks/<功能名称>/` 下创建 README.md：

```markdown
# <功能名称>

> 创建时间：YYYY-MM-DD
> 类型：功能 / 修复 / 优化 / 重构

## 说明

<简要描述这个任务要做什么，为什么要做，2-5 句话即可>

## 开发清单

| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ⏳ 未开始 | <任务描述> | |
| #2 | P1 | ⏳ 未开始 | <任务描述> | |

## 涉及文件

- `path/to/file1.ts` — 修改原因
- `path/to/file2.ts` — 修改原因
```

### 状态标记（与 x-dev 一致）

| 符号 | 状态 | 说明 |
|------|------|------|
| ⏳ | 未开始 | 等待开始 |
| ▶️ | 进行中 | 正在执行 |
| 🟡 | 待测试 | 开发完成，等待验证 |
| 🟢 | 测试通过 | 验证通过 |
| 🔴 | 测试失败 | 需要修复 |
| ✅ | 已完成 | review 确认后标记 |

### 优先级

| 优先级 | 说明 |
|--------|------|
| P0 | 必须完成 |
| P1 | 重要，应当完成 |
| P2 | 可选优化 |

状态流转：`⏳ → ▶️ → 🟡 → 🟢 → ✅`（测试失败时 `🟡 → 🔴 → 修复 → 🟡`）

开发阶段最多到 🟢，✅ 由 review 确认后更新。

## 第三步：逐项开发

按清单顺序（P0 优先）逐项实现：

1. 开始某项前，将其状态改为 `▶️ 进行中`
2. 读懂相关代码，做出修改
3. 修改完成后将状态改为 `🟡 待测试`
4. 验证改动是否正确（运行测试、手动检查等）
5. 通过则改为 `🟢 测试通过`，失败则改为 `🔴 测试失败` 并修复后重新验证
6. 在变更记录中记录关键操作

每完成一项就更新 README.md 中的状态，保持实时同步。

## 第四步：记录变更

在 `dev-pipeline/tasks/<功能名称>/changelog.md` 中记录关键变更：

```markdown
# 变更记录

| 时间 | 操作 | 内容 |
|------|------|------|
| YYYY-MM-DD | 开始开发 | 开始处理 #1：<任务描述> |
| YYYY-MM-DD | 完成开发 | #1 实现完成，测试通过 |
| YYYY-MM-DD | 测试失败 | #2 失败，原因：<原因> |
| YYYY-MM-DD | 修复完成 | #2 修复后重新测试通过 |
```

只记录有价值的信息，不需要事无巨细。重点记录：
- 任务开始和完成
- 测试失败及原因
- 需求调整或方案变更
- 遇到的关键决策

## 第五步：代码审查

所有清单项达到 🟢 后，对本次所有改动执行快速 review。

### 审查范围

检查清单中每一项对应的代码改动，不遗漏。

### 检查维度

1. **功能正确性**：逻辑是否正确、边界条件是否处理、是否满足任务描述的需求
2. **安全与稳定性**：是否引入安全风险（注入、XSS 等）、错误处理是否充分、资源是否正确释放
3. **代码质量**：命名是否清晰、是否有重复代码、复杂度是否合理
4. **兼容性**：改动是否破坏现有功能、API 变更是否向后兼容

### 输出格式

在对话中直接输出 review 结果：

```markdown
## Quick Review

> 审查时间：YYYY-MM-DD
> 涉及文件：N 个
> 改动范围：<简述>

### 问题列表

| # | 严重度 | 文件:位置 | 问题 | 建议 |
|---|--------|-----------|------|------|
| 1 | P0 | `file.ts:32` | 描述 | 修复建议 |
| 2 | P1 | `file.ts:58` | 描述 | 修复建议 |

### 结论

- ✅ 无问题，可以完成 / ⚠️ 有 N 个问题需要修复
```

### 问题处理

- 如果发现 P0/P1 问题：直接修复，更新变更记录，重新 review 修复部分
- 如果只有 P2/P3 问题：列出问题，询问用户是否需要修复
- 全部问题解决后，将清单中对应项的状态从 🟢 更新为 ✅

## 第六步：收尾

所有清单项标记为 ✅ 后：

1. 在变更记录中添加最终记录
2. 向用户汇报完成情况：完成了哪些任务、改了哪些文件、review 结论

## 第七步：提醒提交

开发完成后，提示用户：

```
✅ 开发完成，可提交：

git add .
git commit -m "feat: [一句话描述完成内容]"
```

如果本次涉及多个不相关改动，建议按实际修改分开 commit。

## 注意事项

- 保持轻量：这个流程的核心优势是快。不要在文档上花太多时间，重点是把活干好
- 状态实时更新：每个任务项完成就更新一次，不要等到最后一起改
- 变更记录精简：只记有价值的信息，不需要流水账
- 如果开发过程中发现任务比预期复杂得多，及时告知用户，建议切换到 x-req/x-plan/x-dev 流程
