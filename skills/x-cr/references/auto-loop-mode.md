# 连续开发自动流转模式

本文件是 x-cr 在连续开发循环中的工作规则。SKILL.md 主体会在识别到连续开发上下文时指向这里。

## 什么时候进入这个模式

不依赖任何命令行参数，判断依据是 AI 自身的对话上下文：

- **进入**：当前对话正在执行 `x-dev` 的自动流转规则，或正由 `x-fix` 调用复审同一批文件
- **不进入**：用户直接让 x-cr 审查某个文件 / 目录 / diff，按 SKILL.md 主体的常规流程走完输出报告后停下

理由：x-dev 和 x-fix 在调用 x-cr 前正在执行它们自己的自动流转规则，这个意图是对话上下文的一部分，AI 读取自身上下文即可区分，比依赖参数更可靠。

---

## 独立验证：为什么必须重新对照需求

进入连续开发模式后，x-cr 最重要的一条硬约束是**不信任 x-dev 的自我报告**。

原因：x-dev 在循环中会快速完成任务并报告"✅ 测试通过"，但它的测试可能没覆盖到需求里的某个条款，或者它"按自己的理解"做了需求外的额外改动。这类偏差只有独立验证才能发现。如果 x-cr 跳过这一步，整个循环就失去了质量门控的意义，等同于让 x-dev 自我颁发毕业证。

### 与 SKILL.md 主体"对照文档检查"的关系

SKILL.md 主体的审查流程第 3 步"对照文档检查"已经定义了以需求文档为基准的核心逻辑（定位文档 → 提取约束 → 逐条对照 → 不一致时的处理），两种模式共用这条规则。**连续开发模式不是替换这一步，而是在它之上加几条更严的约束**，下面列出的都是在主体规则之上的补强：

**连续开发模式下的加严项**：

- **需求文档必定存在**：因为 x-dev 是从 task 目录被调用的，task 目录下一定有 `README.md` 和 `dev-checklist.md`。如果连这两个文件都找不到，说明调用链断了，应当停止循环并报错，而不是退化为"没有文档的常规审查"。
- **检查基准更全**：除了主体流程要求的 README.md，还必须同时读 `dev-checklist.md` 中对应任务的描述，两份文档都算作需求来源。
- **不一致处理更严**：
  - 若能在对话上下文或 task 的 changelog.md 中找到"用户授权改动"的痕迹 → 按主体流程处理：标记为"已授权偏差"，不作为 P0，但**必须提醒用户回头同步文档**
  - 若找不到授权痕迹 → 连续开发模式下**仍然停下来问用户**（不要因为在循环中就擅自假设）。x-dev 觉得的"小偏差"可能是需求变化也可能是 bug，连续开发不能代替人做这个判断
  - 确认是"代码错了"的不一致 → 作为 P0 问题计入报告，进入后续自动修复循环
- **镀金和过度扩展的检测**：x-dev 额外实现了需求外的改动也要列出来。循环中镀金会越滚越多，早发现早处理。

**不允许的行为**：

- ❌ 相信 x-dev 的完成声明而不读代码
- ❌ 只审查代码质量而不对照需求
- ❌ 把需求不符的问题降级为 P2/P3 来规避后续的修复循环
- ❌ 在连续循环中因为"怕麻烦用户"就跳过"不一致时询问用户"这一步

## 审查完之后的自动修复循环

x-cr 审查完成并保存报告后，立刻按下面的流程继续推进，不等待用户说"下一步"：

```
1. 统计报告中的 P0 + P1 问题数量

2. 若 P0 + P1 == 0：
   - 在 CR 报告中把 P1.5 / P2 / P3 条目标记为 `⏭auto-loop-skipped`
   - 在 task 的 changelog.md 追加一条汇总：
     `任务 #N 跳过 P1.5×a / P2×b / P3×c（见 cr-report-*.md）`
   - 将当前任务状态在 dev-checklist.md 中改为 ✅ 已完成
   - 返回 x-dev，通知"本任务审查通过，请取下一个任务"

3. 若 P0 + P1 > 0：
   - 读取本任务的 fix_attempts 计数（见下一节）
   - 若 fix_attempts >= 6：
     - 停止自动流转
     - 输出"已达 6 次修复上限"汇报，列出剩余 P0 / P1
     - 返回 x-dev 让其执行"停止条件"汇报
   - 若 fix_attempts < 6：
     - 调用 x-fix，传入当前 CR 报告路径 + 任务编号
     - x-fix 修复完成后会再次调用 x-cr 复审同一批文件
     - 复审时 fix_attempts += 1，回到步骤 1
```

为什么 6 次：足够兜住"修了一个问题又发现相邻问题"这种级联情况，但也不至于在同一个任务上耗尽预算。6 次仍不收敛通常意味着需求或架构本身需要人工介入。

---

## fix_attempts 计数的权威来源

fix_attempts 以**单个任务**维度跟踪，权威来源是 `dev-checklist.md` 对应任务行的"备注"列：

- 首次进入修复循环时，在备注列写入 `[fix:1]`
- 每次 x-fix 完成后回到 x-cr 时 `+1`，更新为 `[fix:2]`、`[fix:3]`……
- 读取时从该列解析 `[fix:N]`，没有则视为 0
- 任务最终改为 `✅ 已完成` 时可移除或保留此标记（保留有利于历史追溯）

CR 报告文件也可能记录这个数字，但**以 dev-checklist.md 为准**，因为任务状态的主档是 dev-checklist.md。

## 报告末尾追加"自动流转状态"

当循环因任何原因停止（通过、达到 6 次上限、或 x-fix 异常）时，在报告末尾追加：

```markdown
---
## 自动流转状态

- 触发模式：连续开发（由 x-dev 调用 / 复审调用）
- fix_attempts：N / 6
- 结论：<已通过 / 达到 6 次上限停止 / 无需修复>
- 下一步：<返回 x-dev 取下一任务 / 停止自动流转，人工介入>
```

这段信息让 x-dev 能够只读报告就判断下一步动作，而不需要重新推理整个循环状态。

## 关键约束速查

| 约束 | 含义 | 为什么 |
|------|------|--------|
| 不信任 implementer 报告 | 必须独立读代码对照需求 | x-dev 的自测盲区需要第二视角兜底 |
| 不等用户确认 | 发现 P0/P1 后直接进入 x-fix | 用户已经在更高层授权连续开发，每步确认会让循环变成手动流程 |
| 不跳过复审 | x-fix 每次修复后必须再次 x-cr 复审 | 修复可能引入新问题或没真的修干净 |
| 不无限循环 | fix_attempts 达到 6 必须停止 | 避免在难收敛问题上耗尽预算 |
| 不降级问题严重度 | 不得把需求不符降级为 P2/P3 | 降级是变相跳过循环，等于作弊 |
| 不在循环里处理 P1.5+ | 架构级问题标记后跳过，进入汇总 | 架构修改不适合在自动循环中反复试探 |
