## 状态符号含义

| 符号 | 含义 |
|------|------|
| ✅ | 通过 — 该检查项无问题 |
| ⚠️ | 需讨论 — 有疑点或灰色地带，由人类决策 |
| ❌ | 问题 — 明确的缺陷，需要修复 |
| ⏭ | 跳过 — 如 `⏭auto-loop-skipped`，用于连续开发模式下跳过 P1.5/P2/P3 |

## 问题详情

### B1: 密码校验失败未清理 session
**来源**: 通用检查项
**文件**: `src/auth/login.ts`
**位置**: 第 47 行
**严重程度**: P0

**问题描述**：密码校验失败时直接 `return` 而未调用 `session.clear()`，导致下一次登录尝试会带入旧的 session state。

**影响**：用户密码输错一次后再登录会进入一个"半登录"状态，session 中混入残留字段，下游判断可能失效。

**修复建议**：在 return 前加 `await session.clear()`，或改用 `try/finally` 确保失败路径也会清理。

### B2: async createSession 缺少错误处理
**来源**: 语言专项检查项（TypeScript async 规则）
**文件**: `src/auth/session.ts`
**位置**: 第 22 行
**严重程度**: P1

**问题描述**：`createSession` 是 async 函数，内部 `await db.insert(...)` 没有 try/catch，数据库异常会直接抛成未处理 Promise。

**影响**：生产环境出现 DB 抖动时会被当成 500 内部错误暴露给客户端。

**修复建议**：包一层 try/catch，把 DB 错误转成业务错误码返回；或在上层 Express 的错误中间件中统一兜底。

### B3: 密码错误返回 reason 可能助长撞库
**来源**: 通用检查项
**文件**: `src/auth/login.ts`
**位置**: 第 33 行
**严重程度**: P1（需讨论）

**问题描述**：登录失败时响应中包含 `reason: 'password'`，攻击者可据此判断用户名是否存在。

**影响**：撞库攻击效率提升。但如果产品方希望给用户更友好的错误提示，这是权衡问题。

**修复建议**：若合规要求严，统一返回通用错误；若产品优先体验，保留但加限流。建议与产品方确认。

## 最终结论

- **不建议合并**，有 1 个 P0 必须修复
- 必须修复项：
  - B1（P0）密码校验失败清理 session
  - B2（P1）createSession 错误处理
- 建议修复项：
  - B3（P1 需讨论）撞库风险与产品体验的权衡
  - 补充登录失败分支的单元测试
