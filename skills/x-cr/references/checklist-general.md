# 正确性检查清单

x-cr 只检查会影响软件正确性的事项。每一项结论使用 `通过 / 问题 / 需讨论` 三态，方便下游把明确问题交给 x-fix，把需讨论项交给用户决策。

## P0：已确认正确性错误

P0 表示当前代码已经高置信地产生错误结果或核心路径失败。

- 用户报告的现象能映射到确定代码路径。
- 原始 spec 明确要求 A，当前实现做了 B。
- 核心流程存在不可能成功的调用链。
- 条件判断会进入错误分支。
- 数据读写、状态更新、序列化/反序列化会产生错误结果。
- 鉴权、权限、租户隔离、数据边界存在确定绕过。
- 迁移、删除、写入操作可能破坏数据。
- 对抗性输入、状态、依赖失败、权限或并发场景能触发确定错误结果。

P0 必须包含：

- 触发条件。
- 失败路径。
- spec 或用户期望。
- 当前实现证据。
- 反证检查结果。

## P1：生产正确性风险

P1 表示当前证据支持“线上可见失败”，还需要修复或用户确认优先级。

- 边界输入会导致异常、空结果或错误分支。
- 部分初始化、重复调用、资源关闭后复用会导致状态错误。
- 错误处理路径会吞掉失败，导致调用方认为成功。
- 并发、重试、缓存、幂等逻辑存在中高置信风险。
- 配置缺省值、环境变量、数据形状变化会改变行为。
- 测试覆盖了 happy path，关键失败路径缺口会让错误逃逸。
- 对抗性假设有调用链证据支持，仍缺复现、测试或日志把置信度提升到已确认。

P1 必须说明：

- 哪个用户路径或系统路径会受影响。
- 当前证据如何支持风险。
- 还缺哪条证据能把置信度提高到已确认。

## P2：正确性证据缺口

P2 表示当前无法完成正确性判断，但缺口会影响后续修复或验收。

- 原始 spec 缺少关键边界、状态、错误路径或验收标准。
- 测试没有断言核心契约，只证明代码能跑。
- 日志缺少定位信息，无法区分候选根因。
- dev-report 或 PR 描述没有覆盖用户可见结果。
- 模块公开入口和调用方契约缺少文档。
- 对抗性候选缺少触发条件、调用链、失败路径或反证检查。

P2 输出应给出最短补证路径：

- 需要补哪条测试。
- 需要补哪条日志。
- 需要确认哪条 spec。
- 需要读取或运行哪条命令。

## 退出范围

这些内容归属其他 skill 或人工设计讨论：

- 命名、格式、排版。
- 纯代码风格。
- 单纯抽象偏好。
- 没有错误路径证据的架构建议。
- 性能优化建议，除非它会导致超时、资源耗尽或用户可见失败。

## Spec 对照要求

每个 P0/P1 问题都要给出 spec 关系：

| 分类 | 判定 |
|------|------|
| 原始 spec 不一致 | 代码行为和明确 spec 冲突 |
| 实现过程偏移 | spec 清楚，实现漏做、做偏、回归 |
| spec 缺口 | spec 缺少关键约束 |
| 环境/数据问题 | 代码契约成立，环境、配置或数据异常 |
| 证据不足 | 关键证据缺失 |
