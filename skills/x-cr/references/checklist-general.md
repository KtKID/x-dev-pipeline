# 通用检查清单

按 P0 → P3 由重到轻的顺序组织。每一项的结论必须是 `通过 / 问题 / 需讨论` 三选一，理由是这样下游可以直接把 "问题" 汇总为必修项，把 "需讨论" 单独留给人类决策。

---

## P0：功能正确性 / 致命缺陷 {#p0}

这一级关注的是"代码跑起来结果就是错的"这种级别的问题。任何一条命中都意味着改动不应该合并。

- 功能逻辑是否明显错误
- 调用链是否存在不可能成功的调用
- 条件判断是否导致错误分支执行
- 数据读写是否可能直接产生错误结果
- 核心流程是否存在明显中断风险

**spec 合规子项**：

这一项的检查逻辑由 SKILL.md 主体的审查流程第 3 步"对照文档检查"统一定义，两种模式都走同一条流程 —— 定位需求文档、提取约束、逐条对照、不一致时先回忆上下文再询问用户。**不要在这里重复实现**，遵循 SKILL.md 主体的规则即可。

连续开发模式下这一步是硬性的额外约束：

- 需求文档必定存在（task 目录的 `README.md` + `dev-checklist.md`），找不到就停止循环
- 发现"代码与文档不一致且无用户授权痕迹"时，仍然要停下来问用户，不要因为在自动循环里就跳过
- 详见 `auto-loop-mode.md` 的"独立验证"小节

---

## P1：稳定性 / 安全性 / 数据风险 {#p1}

P1 关注的是"现在能跑，但生产环境会出事"。安全漏洞、资源泄漏、并发问题通常不会在开发环境被触发，但一旦上线就是事故。

- 错误处理是否缺失
- 边界输入是否可能导致异常
- 是否存在明显安全风险（注入、XSS、权限绕过、敏感信息泄漏等）
- 是否存在数据一致性风险
- 是否存在资源释放或生命周期管理风险
- 是否存在明显并发或竞态风险

---

## P1.5：架构设计 / 设计原则 {#p15}

P1.5 不是"错"，而是"长期会变成错"。这一级的检查帮助发现会滚雪球的结构问题，值得单独列一级是因为它往往需要跨文件重构，不能在一次循环里修完。

**SOLID 原则**：

- 单一职责（SRP）：每个模块是否只负责一件事
- 开闭原则（OCP）：是否对扩展开放、对修改封闭
- 里氏替换（LSP）：子类是否能替换父类而不破坏功能
- 接口隔离（ISP）：是否使用小而专的接口
- 依赖倒置（DIP）：是否依赖抽象而非具体实现

**模块化设计**：

- 层次是否清晰（表现层 / 业务层 / 数据层分离）
- 模块间是否低耦合（依赖是否单向、是否有循环依赖）
- 公共逻辑是否抽取到独立模块
- 是否存在重复代码或重复逻辑

**数据流设计**：

- 数据流向是否清晰
- 状态管理是否合理
- 是否存在不必要的数据复制

**扩展性设计**：

- 新增功能是否需要大量修改现有代码
- 是否预留了扩展点
- 配置是否外置

---

## P2：可维护性 / 可读性 / 测试质量 {#p2}

P2 的问题不影响当前功能，但会让未来改动这段代码的人变慢。命名、重复、测试覆盖属于这一级。

- 命名是否清晰表达意图
- 是否有重复实现
- 模块职责是否清晰
- 是否存在过度复杂逻辑
- 测试是否覆盖关键路径
- 是否缺少必要注释或说明

---

## P3：优化建议 / 边界增强 {#p3}

P3 是"可以更好"的层级，不是"必须改"。列出来是为了让开发者看到后续演进方向，但不应该为了修 P3 阻塞合并。

- 空数据和异常输入处理是否充分
- 极端场景是否考虑
- 性能热点是否有明显隐患
- 是否存在更合理的抽象或拆分方式
- 是否有后续演进风险
