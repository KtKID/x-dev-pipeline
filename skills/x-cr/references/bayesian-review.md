# 贝叶斯根因调查法

本文件定义 x-cr 的调查方式。目标是把“看起来像问题”的直觉判断转成“根因假设经过证据更新后的置信度”。

## 公式

```text
P(H | E) = P(E | H) * P(H) / P(E)
```

在 x-cr 中：

| 符号 | 含义 |
|------|------|
| H | 候选根因，例如“配置读取路径错了” |
| E | 证据，例如日志、测试失败、代码路径、diff、spec |
| P(H) | 先验：看到证据前该根因的可能性 |
| P(E | H) | 如果该根因成立，看到当前证据的可能性 |
| P(H | E) | 看到证据后，该根因成立的置信度 |

## 定性等级

| 等级 | 含义 |
|------|------|
| 低 | 当前证据明显削弱该根因 |
| 中 | 有支持证据，也有缺口或反证 |
| 高 | 多条证据支持，反证检查未推翻 |
| 已确认 | 有复现、明确调用链、直接 spec 冲突或确定失败路径 |

## 两种入口

### 已知问题入口

用户已经给出现象，例如“xx 不太对”。调查顺序：

1. 抽取用户现象、期望行为、实际行为。
2. 把用户现象作为第一证据 E0。
3. 列出候选根因 H1-Hn。
4. 逐步读取日志、代码、测试、配置、spec，更新每个根因置信度。
5. 找到最高置信根因后，对照原始 spec。

### 模块 review 入口

用户让 review 某个模块。调查顺序：

1. 找模块入口、公开 API、状态流、数据流和原始 spec。
2. 生成会导致错误结果的候选根因，并主动构造对抗性假设。
3. 读取调用链、边界处理、错误路径、测试断言。
4. 只把高置信或中高置信正确性风险写入报告。

## 对抗性候选生成

x-cr 在 H 列表中优先纳入会击穿实现或结论的假设：

- 输入：空值、空集合、非法类型、超大输入、路径穿越、特殊字符。
- 状态：未初始化、部分初始化、已关闭资源复用、跨请求状态污染。
- 依赖：外部服务超时、空响应、非法 JSON、schema 漂移、部分失败。
- 权限：凭据缺失、用户拒绝授权、越权输入、租户边界混淆。
- 并发：重复提交、重试、乱序完成、缓存过期、幂等性破坏。
- 测试：happy path-only、自我 mock、用业务逻辑计算期望值、fixture 过拟合。

每个对抗性候选需要写清触发条件、预期失败路径、可能影响和最短取证动作。没有代码路径、spec、测试、日志或 diff 证据支撑的候选，进入 P2 证据缺口或已排除假设。

## 证据分类

支持证据：

- 用户现象与某条代码路径一致。
- 日志、栈、测试失败指向同一函数或状态。
- diff 删除 guard、改变分支、改变数据映射。
- spec 要求 A，代码实现 B。
- 调用链能传入风险输入或进入风险状态。

削弱证据：

- 上游已经保证输入条件。
- 类型、schema、运行时校验覆盖该风险。
- 测试覆盖目标路径且断言契约结果。
- 配置默认值或迁移逻辑已经处理该状态。
- spec 明确允许当前行为。

中性证据：

- 只说明文件被改动。
- 只说明函数存在。
- 只说明测试通过，且未确认测试覆盖目标路径。

## 根因分类

| 分类 | 使用条件 |
|------|----------|
| 原始 spec 不一致 | spec 明确，当前行为冲突 |
| 实现过程偏移 | spec 明确，实现漏做、做偏、回归、半成品 |
| spec 缺口 | spec 缺少关键边界、状态、错误路径或验收标准 |
| 环境/数据问题 | 代码契约成立，配置、环境、数据形状、迁移状态异常 |
| 证据不足 | 缺少复现、日志、调用链、spec 或关键文件 |

## 输出位置

报告字段、问题详情、示例内容统一放在 `report-template.md`。本文件只定义调查方法和判定规则。

## 判定规则

- `已确认` 和 `高` 置信度可以进入 P0/P1。
- `中` 置信度进入 P1 需讨论或 P2 证据缺口。
- `低` 置信度写入“已排除假设”。
- 测试通过只能作为辅助证据；需要确认测试覆盖目标路径和契约断言。
