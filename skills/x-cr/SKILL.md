# x-cr 代码审查

x-cr 负责结构化代码审查，读取代码 → 按严重度分级 → 输出可追溯的报告。x-cr 不修改代码，修改由下游的 x-fix 处理；x-cr 不决定合不合并，决定由人或调用者根据报告做出。这种分工让每个 skill 的职责单一，才让连续开发循环（x-dev → x-cr → x-fix → x-cr → x-dev）能跑通。

---

## 两种模式

x-cr 有两种工作模式，AI 根据当前对话上下文自主判断，不依赖任何命令行参数。

| 模式 | 何时进入 | 审查结束后的动作 |
|------|---------|-----------------|
| **手动审查** | 用户直接让你 review 某段代码 / 文件 / diff / PR / 粘贴片段 | 保存报告后停下，后续由用户决定 |
| **自动流转审查** | 当前对话正在执行 x-dev 的连续开发规则，或 x-fix 修完后回来复审 | 保存报告后，立刻按 `references/auto-loop-mode.md` 的规则继续：决定是关闭任务、触发 x-fix、还是在 6 次上限处停下 |

自动流转模式有一条额外硬约束：**不信任 x-dev 的自我报告**。这一模式下，P0 会在"功能正确性"检查之前追加一项"spec 合规检查"，逐条对照任务需求。详见 `references/auto-loop-mode.md`。

---

## 审查流程

两种模式共用同一条流程：

```
定位需求文档 → 收集变更 → 识别语言 → 对照文档检查 → 通用检查 → 语言专项检查 → 按严重度汇总 → 渲染报告 → 保存报告
```

这几步按顺序进行的原因是：**需求文档定义了代码应该长什么样，所有后续检查都是以它为基准**。没有需求文档作为参照，"功能是否正确" 只能凭感觉判断；有了文档，偏差就有客观依据。语言识别决定后续加载哪份专项清单；通用检查提供严重度分级的基线，语言检查的问题才能对齐相同的分级标准。

### 0. 定位需求文档

审查前先尝试找到这次改动对应的需求文档。候选顺序：

1. **当前 task 目录里的 README.md / design.md / req.md**（如果 x-cr 是从 x-dev / x-qdev / x-fix 流程中被调用，task 目录通常是已知的）
2. **用户在请求中显式指定的文档路径**
3. **从被审查代码的路径能推断出的 task 目录**（例如代码属于某个 feature branch 的工作目录）
4. **找不到**：退回常规审查，直接对代码本身做通用检查 + 语言专项检查，不强求有文档

找到文档后，读取并提取其中的关键约束：核心目标、包含/不包含范围、功能模块拆分、显式列出的"必须/应当/shall/must/will"条款。这些将作为下一步"对照文档检查"的基准。

### 1. 收集变更

可接受的输入形式：git diff、单个文件、多个文件、目录、用户粘贴的代码片段、指定模块名。如果用户没给明确范围，优先看用户提到的文件或最明显相关的几个文件，不要擅自扩大审查面。

### 2. 识别语言

识别顺序：文件扩展名 → 目录结构 → 语法特征 → 用户显式说明。无法判断时，只跑通用检查，并在报告里注明未加载语言专项配置。

### 3. 对照文档检查（仅当找到需求文档时）

这一步的目的是回答一个问题：**代码是不是按文档说的那样实现的？**

逐条对照第 0 步提取出的文档约束，针对每一条：

- 代码里有没有对应实现？
- 行为是不是文档描述的那样？
- 有没有做文档里没写的额外事情（镀金 / 过度扩展）？
- 有没有漏掉文档明确要求的事情？

**当发现代码与文档不一致时，不要立刻判为 P0 问题，先执行下面的"不一致处理"流程**：

#### 不一致处理流程

1. **先回忆对话上下文**：在当前对话（以及最近的 task changelog.md）里搜索是否出现过"用户让改了实现"、"讨论后决定不按文档做"、"这个需求已经变了" 这类痕迹。搜索线索包括：
   - 用户曾明确说"改一下这里"、"这个不要了"、"换个方式"
   - 任务的 changelog.md 记录了"需求变更"、"方案调整"条目
   - 对话里曾经讨论过某条需求并达成了不同的结论

2. **找到了授权痕迹**：说明这次不一致是用户知情并同意的。在报告中单独列一节"已授权的代码与文档偏差"，列出每条偏差 + 对应的对话/changelog 引用，不作为 P0 问题。同时提醒用户"文档尚未同步，建议回头更新需求文档以免下次审查再次触发"。

3. **找不到任何授权痕迹**：这是一个**未决的不一致**，不要擅自假设任何一方正确。**停下报告生成**，向用户发问：
   ```
   ⚠️ 发现代码与需求文档不一致，但在对话和 changelog 里没有找到修改授权：

   文档说：<文档原文>
   代码是：<代码实际行为>
   位置：<文件:行号>

   请确认应该：
   （A）改代码以符合文档
   （B）改文档以符合代码（说明修改原因）
   （C）这是误判，代码和文档其实一致

   确认后我继续生成审查报告。
   ```

   用户回答后按以下规则处理：
   - 选 A → 在报告中把这条作为 P0（代码实现不符合需求）
   - 选 B → 当场更新文档（或请求用户更新），然后把这条记入"已授权偏差"
   - 选 C → 说明自己理解有误，从报告中移除这条并继续

**为什么这样处理**：代码与文档不一致有两种成因，一种是"代码错了"（bug），一种是"文档过时了"（需求变化没同步）。这两种需要完全不同的处理方式 —— 前者要修代码，后者要改文档。没有用户确认就主观判断任何一方"错了"，会导致 x-fix 在连续开发循环里去修一个其实不用修的"问题"，或反过来让真正的 bug 被当成"文档过时"放过。

### 4. 执行通用检查

通用检查清单详见 `references/checklist-general.md`。每一项都要给出 `通过 / 问题 / 需讨论` 三种结论之一 —— 这三分法的存在是为了让下游能把结果自动分成"必修"、"需人类决策"、"通过"三堆。

| 严重度 | 主题 | 详见 |
|-------|------|------|
| P0 | 功能正确性 / 致命缺陷 | `references/checklist-general.md#p0` |
| P1 | 稳定性 / 安全性 / 数据风险 | `references/checklist-general.md#p1` |
| P1.5 | 架构设计 / 设计原则 | `references/checklist-general.md#p15` |
| P2 | 可维护性 / 可读性 / 测试质量 | `references/checklist-general.md#p2` |
| P3 | 优化建议 / 边界增强 | `references/checklist-general.md#p3` |

自动流转模式下，`checklist-general.md#p0` 还会要求你在常规 P0 检查前先做一遍 spec 合规子检查。

### 5. 执行语言专项检查

按识别到的语言加载对应的 reference 文件，把专项问题追加在通用问题之后。语言专项只检查语言特有问题，不重复通用检查项 —— 重复会让报告变得啰嗦，也让严重度分级失去区分度。

| 语言 | 文件 |
|-----|------|
| TypeScript | `references/lang-ts.md` |
| JavaScript | `references/lang-js.md` |
| C# | `references/lang-csharp.md` |

其他语言（Python / Java / Go / Rust 等）暂时没有专项清单，只跑通用检查，在报告中注明"未加载语言专项规则"。需要时可按需补充。

变更跨多种语言时：先跑一次通用检查，然后对每种语言分别加载专项清单，每条问题都标注来源语言。

### 6. 渲染报告

完整模板和一个真实的登录 bug 填充示例放在 `references/report-template.md`。精简版：每个严重度各一张汇总表，每个问题还要有一段 `B<N>` 详情块（文件、行号、描述、影响、修复方向）。结尾给一个汇总表和"可合并 / 不建议合并 / 需讨论"的最终结论。

### 7. 保存报告

命名规则：`cr-report-YYYYMMDD-HHmmss.md`（本地时间，时间戳内部无分隔符）。

保存位置遵循"跟随调用者 task 目录"的软规则：

- 如果 x-cr 是从 x-dev / x-qdev / x-fix 的流程中被调用、且当前有 task 目录（无论这个目录在 `.claude/tasks/`、`dev-pipeline/tasks/` 还是别的位置），保存到那个目录
- 如果被审查的代码明显属于某个可通过路径或上下文推断的 task 目录，保存到那个目录
- 否则，询问用户保存位置，或退一步保存到被审查代码所在的目录

保存后在回复末尾追加一行：

```
📄 报告已保存：<完整路径>
```

## 全局审查原则

这些原则适用于两种模式，优先级高于任何具体检查项：

- **基于实际变更**：每一条问题都要绑定到具体的文件和行号，不做无关泛化。为了凑数量而加的"一般建议"会稀释真正的问题，让后来人看报告时失去注意力。
- **逐项给结论**：每个维度都要明确给出 `通过 / 问题 / 需讨论`。"问题" 必须可定位，"需讨论" 表示这是你不想独自下结论的判断题。
- **可定位**：文件路径、行号或代码区域、严重度、检查项名称、来源类型（通用 vs 语言专项）缺一不可。缺了其中任何一项，下游就没法对这条问题采取行动。
- **避免伪问题**：某个维度没发现问题就写"未发现明显问题"，不要硬编一个弱问题凑数。凑数会让整份报告的可信度整体下降。
- **优先真实风险**：一个真正阻塞功能的 P0 比十个鸡毛蒜皮的 P2 更有价值。问题排序时让"会导致功能错 / 数据错 / 稳定性下降 / 安全漏洞 / 显著维护成本"的放在最前面。

## x-cr 的边界

x-cr 只做"读 + 评"。它不：

- 自动修改或给代码打补丁（那是 x-fix 的职责）
- 自动提交或推送
- 用语言专项规则替代通用检查

保持职责单一是自动流转模式能跑通的前提：x-cr 产出报告 → x-fix 读报告并修复 → 交回 x-cr 复审。如果任何一方越界，这条链路就断了。
