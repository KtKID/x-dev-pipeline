# x-cr 代码审查框架

## 核心定位

x-cr 是代码审查框架 skill。

x-cr 负责：
1. 读取代码变更、文件或目录
2. 识别主要语言和代码场景
3. 执行通用检查项
4. 加载对应语言专项配置并执行专项检查
5. 输出统一格式的 review 报告

x-cr 不负责：
- 自动修改代码
- 自动提交代码
- 自动生成修复 patch
- 用语言专项规则替代通用检查

---

## 审查流程

按以下顺序执行，不可跳步：

1. 获取变更内容
2. 识别主要语言
3. 执行通用检查项
4. 根据语言加载专项配置
5. 执行语言专项检查项
6. 汇总问题并按严重程度分级
7. 输出统一格式的 review 报告
8. 将review报告保存到对应.claude/task/'具体功能'对应目录中，如果找不到则询问用户是否创建报告

如果无法明确识别语言：
- 只执行通用检查项
- 在报告中明确说明未加载语言专项配置

---

## 语言配置加载规则

优先根据以下线索识别语言：
1. 文件扩展名
2. 目录结构
3. 代码语法特征
4. 用户显式说明

识别后读取对应配置文件：

- TypeScript：`references/lang-ts.md`
- JavaScript：`references/lang-js.md`
- C#：`references/lang-csharp.md`
- Python：`references/lang-python.md`
- Java：`references/lang-java.md`
- Go：`references/lang-go.md`
- Rust：`references/lang-rust.md`

如果一个变更包含多种语言：
- 先执行通用检查
- 再对每种语言分别加载对应专项配置
- 报告中标明问题来源语言

语言配置文件只定义专项检查项，不重复定义通用检查项。

## 通用检查项

默认必须执行以下通用检查项。

### P0：功能正确性 / 致命缺陷

检查：
- 功能逻辑是否明显错误
- 调用链是否存在不可能成功的调用
- 条件判断是否导致错误分支执行
- 数据读写是否可能直接产生错误结果
- 核心流程是否存在明显中断风险

结论要求：
- 每项都必须给出“通过 / 问题 / 需讨论”

---

### P1：稳定性 / 安全性 / 数据风险

检查：
- 错误处理是否缺失
- 边界输入是否可能导致异常
- 是否存在明显安全风险
- 是否存在数据一致性风险
- 是否存在资源释放或生命周期管理风险
- 是否存在明显并发或竞态风险

结论要求：
- 每项都必须给出“通过 / 问题 / 需讨论”

---

### P1.5：架构设计 / 设计原则

检查：

**SOLID 原则**：
- 单一职责（SRP）：每个模块是否只负责一件事
- 开闭原则（OCP）：是否对扩展开放、对修改封闭
- 里氏替换（LSP）：子类是否能替换父类而不破坏功能
- 接口隔离（ISP）：是否使用小而专的接口
- 依赖倒置（DIP）：是否依赖抽象而非具体实现

**模块化设计**：
- 层次是否清晰（表现层 / 业务层 / 数据层分离）
- 模块间是否低耦合（依赖是否单向、是否有循环依赖）
- 公共逻辑是否抽取到独立模块
- 是否存在重复代码或重复逻辑

**数据流设计**：
- 数据流向是否清晰
- 状态管理是否合理
- 是否存在不必要的数据复制

**扩展性设计**：
- 新增功能是否需要大量修改现有代码
- 是否预留了扩展点
- 配置是否外置

结论要求：
- 每项都必须给出"通过 / 问题 / 需讨论"

---

### P2：可维护性 / 可读性 / 测试质量

检查：
- 命名是否清晰表达意图
- 是否有重复实现
- 模块职责是否清晰
- 是否存在过度复杂逻辑
- 测试是否覆盖关键路径
- 是否缺少必要注释或说明

结论要求：
- 每项都必须给出“通过 / 问题 / 需讨论”

---

### P3：优化建议 / 边界增强

检查：
- 空数据和异常输入处理是否充分
- 极端场景是否考虑
- 性能热点是否有明显隐患
- 是否存在更合理的抽象或拆分方式
- 是否有后续演进风险

结论要求：
- 每项都必须给出“通过 / 问题 / 需讨论”

---

## 输入来源

支持以下输入形式：
- git diff
- 单个文件
- 多个文件
- 目录
- 用户粘贴的代码片段
- 指定模块或子系统

如果用户没有给出明确范围：
- 默认优先审查用户提到的文件或变更
- 若范围仍不清楚，聚焦最可能相关的文件

---

## 审查规则

### 1. 必须基于实际变更审查
只针对变更内容、相关上下文和直接影响范围给出问题，不做无关泛化建议。

### 2. 必须逐项给结论
每个检查维度都必须给出明确结论：
- 通过
- 问题
- 需讨论

### 3. 问题必须可定位
每个问题都必须尽量标注：
- 文件
- 行号或代码位置
- 严重程度
- 所属检查项
- 来源类型（通用项 / 语言专项项）

### 4. 避免伪问题
不要为了凑数量而输出低价值问题。
如果没有发现问题，应明确写“未发现明显问题”。

### 5. 优先真实风险
优先指出会导致：
- 功能错误
- 数据错误
- 稳定性下降
- 安全问题
- 维护成本显著上升
的缺陷。

## 语言专项配置使用方式

当识别到语言后，读取对应语言配置，并在通用检查后追加专项检查。

专项检查输出要求：
1. 只检查对应语言特有问题
2. 不重复通用检查项
3. 在报告中标注问题来源为“语言专项检查项”
4. 若语言配置中没有命中的问题，可写“该语言专项项未发现明显问题”

## 报告保存

完成审查后，将报告保存为 Markdown 文件：

### 保存路径

按以下优先顺序确定保存目录：

1. **有关联 task 目录**：若审查的代码属于某个 `.claude/tasks/<功能名称>/` 目录（通过用户指定、审查路径或 x-dev/x-qdev 上下文推断），保存到该目录下
2. **无关联 task**：保存到被审查代码所在目录

### 文件命名

```
cr-report-YYYYMMDD-HHmmss.md
```

示例：`cr-report-20260312-143025.md`

时间取当前本地时间，格式为 `年月日-时分秒`（无分隔符）。

### 保存后输出

保存成功后，在回复末尾追加一行：

```
📄 报告已保存：<完整文件路径>
```
