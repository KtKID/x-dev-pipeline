# x-fix CR 报告修复执行框架

## 核心定位

x-fix 是 x-cr 的配套修复 skill。

x-fix 负责：
1. 定位 x-cr 生成的 CR 报告
2. 解析所有 ❌/⚠️ 问题条目
3. 逐条判断有效性并执行代码修复
4. 更新报告中的状态标记
5. 在报告末尾追加修复备注汇总

x-fix 不负责：
- 生成新的 CR 报告（请使用 x-cr）
- 修改 P3 优化建议（默认跳过，除非用户明确要求）
- 在无报告的情况下自行判断哪些代码需要修复

---

## 执行流程

按以下顺序执行，不可跳步：

### 步骤 1：定位报告

报告查找优先级：
1. 用户在指令中显式指定路径 → 直接使用
2. 当前工作目录下最新的 `cr-report-*.md`
3. `.claude/tasks/` 各子目录下最新的 `cr-report-*.md`

**无报告时**：立即停止，输出：
```
❌ 未找到 CR 报告，请先运行 x-cr 生成报告。
```

**报告格式异常时**：立即停止，输出：
```
❌ 报告结构不符合预期，无法解析（期望包含"审查结论"表格和"问题详情"章节）。
```

---

### 步骤 2：解析问题清单

从报告中提取所有问题条目：

1. 扫描「审查结论」各 P 级表格，收集状态为 ❌ 或 ⚠️ 的行
2. 提取每行的：严重程度（所属 P 级）、文件:行号、问题描述
3. 扫描「问题详情」章节，提取对应 Bn 编号的完整信息（文件、行号、严重程度、问题描述、修复建议）
4. 将审查结论表格条目与问题详情 Bn 条目关联

**已含 `✅已修复` 或 `➖无需修复` 或 `⏭已跳过` 标记的条目**：跳过（已被之前的 x-fix 处理过）。

---

### 步骤 3：确定处理范围

默认处理规则：
- **P0**：必须处理
- **P1**：必须处理
- **P1.5**：必须处理
- **P2**：处理
- **P3**：默认**跳过**

---

### 步骤 4：汇报计划

在开始修复前，向用户汇报：

```
📋 修复计划

将处理：
- B1 (P0) xxx.ts:32 — 条件判断导致错误分支执行
- B2 (P1) xxx.ts:80 — Promise 错误未被捕获

将跳过（P3）：
- B4 (P3) zzz.ts:90 — 建议使用更合理的抽象

共 N 条需处理，M 条跳过。
```

如用户希望调整（包含或排除某些条目），先按用户指示调整范围，再继续。

---

### 步骤 5：逐条处理

**串行处理**，每条处理完立即更新报告，不批量堆积。

#### 每条的处理逻辑

**a) 读取上下文代码**

读取问题所在文件，定位到报告中标注的行号，读取前后 10~20 行的上下文。

**行号偏移处理**：
- 若行号处内容与报告描述不匹配，尝试通过代码特征（函数名、关键词）重新定位
- 若重新定位失败，暂停该条目，询问用户

**文件不存在处理**：
- 记录为 `➖无需修复`，备注"文件不存在，可能已重构或删除"
- 继续下一条

**b) 判断处置情形**

| 情形 | 含义 | 操作 |
|------|------|------|
| A | 确认是问题，按报告中的修复建议修复 | 执行代码修改 |
| B | 确认是问题，但有更合适的修复方案 | 执行代码修改，备注中说明原因 |
| C | 误报，实际不需要修复 | 不修改代码，记录 `➖无需修复` |
| D | 影响较大或无法确定，需用户确认 | 暂停，向用户描述情况并征询意见 |

**多条问题互相影响**：
- 若修复 Bn 时已连带解决了 Bm 的问题，则在处理 Bm 时记录 `✅已修复`，备注"已被 Bn 修复消除"，不重复修改代码

**c) 执行代码修改（情形 A/B）**

- 使用精准的 Edit 操作，只改动必要的代码
- 不引入无关的格式调整或重构
- 改完后立即继续

**d) 更新报告**

每条处理完后立即更新报告：

1. 在「审查结论」表格中，将该行的状态标记替换：

   | 原状态 | 修复后 | 无需修复后 |
   |--------|--------|-----------|
   | ❌ | `✅已修复` | `➖无需修复` |
   | ⚠️ | `✅已修复` | `➖无需修复` |

2. 在对应 `### Bn:` 条目末尾追加处置信息：

   ```markdown
   **处置结果**：✅已修复 / ➖无需修复 / ⏭已跳过
   **处置说明**：[简述处置原因或修复方案]
   ```

---

### 步骤 6：追加修复备注

所有条目处理完成后，在报告文件末尾追加以下内容：

```markdown
---
## 修复备注
> 修复执行时间：YYYY-MM-DD HH:mm

| # | 严重程度 | 文件 | 处置结果 | 修复方式 | 备注 |
|---|----------|------|----------|----------|------|
| B1 | P0 | `xxx.ts:32` | ✅已修复 | 同方案 | |
| B2 | P1 | `xxx.ts:80` | ✅已修复 | 改用 try-catch 包裹整个 async 块 | 报告建议范围偏小 |
| B3 | P2 | `yyy.ts:15` | ➖无需修复 | - | 误报：上游已有空值保护 |
| B4 | P3 | `zzz.ts:90` | ⏭已跳过 | - | P3 优化建议，暂缓 |

### 汇总
| 处置类型 | 数量 |
|----------|------|
| ✅ 已修复 | X |
| ➖ 无需修复（误报） | X |
| ⏭ 已跳过（P3） | X |
```

---

### 步骤 7：输出汇报

在对话中输出修复结果汇总：

```
✅ 修复完成

- 已修复：X 条
- 无需修复（误报）：X 条
- 已跳过（P3）：X 条
- 修改文件：X 个（列出文件路径）
```

---

### 步骤 8：路径提示

在输出末尾追加：

```
📄 报告已更新：<完整文件路径>
```

## 修复规则

### 1. 只修复已标识的问题
仅针对 CR 报告中明确标注的问题位置进行修改，不做报告范围外的调整。

### 2. 最小化修改原则
只改动解决问题所必需的代码，不调整无关格式、不做额外重构。

### 3. 保持代码风格一致
修复代码应与周围代码保持相同的风格、命名规范和缩进方式。

### 4. 串行处理，实时更新
逐条处理，每条完成后立即更新报告，不等到全部完成再统一写入。

### 5. 不擅自跳过 P0/P1
P0 和 P1 问题必须处理，不可因"理解复杂"为由跳过。如确实无法判断，以情形 D 征询用户意见。

---

## 边界情况处理汇总

| 情况 | 处置方式 |
|------|----------|
| 无 CR 报告 | 停止，提示先运行 x-cr |
| 报告格式异常 | 停止，提示结构不符合预期 |
| 文件不存在 | 记录 `➖无需修复`，备注"文件不存在" |
| 行号偏移但能重新定位 | 自动定位后继续处理 |
| 行号偏移且无法定位 | 暂停该条，询问用户 |
| 多条问题互相影响 | 修复时一并处理，后续条目记录已被消除 |
| P3 条目 | 默认跳过，记录 `⏭已跳过` |
| 报告中已有修复标记 | 跳过该条，不重复处理 |
