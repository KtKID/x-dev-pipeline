# 模式 1：用户 Bug 修复

> 当用户直接描述 bug 或问题现象时执行本流程。
> 不需要也不应读取任何 CR 报告。

## 执行流程

### 步骤 1：理解 Bug

从用户描述中提炼：
- 问题现象（用户实际看到了什么）
- 复现步骤（如有）
- 影响范围（哪个功能、哪个用户场景）
- 关键约束（日志、环境、版本等如有）

### 步骤 2：定位根因

**不猜测，必须读代码。**

1. 根据问题现象推断最可能的代码位置
2. 使用 Grep / Read 定位相关代码
3. 阅读上下文，理解代码逻辑
4. 确认根本原因（不是表象，是根因）
5. 向用户简述根因并确认方向正确后，再动手修复

**无法定位时**：
- 列出已排查的路径和原因
- 向用户请求更多信息（日志、堆栈、相关配置等）
- 不可随意猜测修复

### 步骤 3：修复

- 使用精准的 Edit 操作，只改动必要的代码
- 不引入无关的格式调整或重构
- 改完后说明修复内容

### 步骤 4：验证

- 简要说明如何验证修复有效
- 如有相关测试用例，指出应运行哪些测试

### 步骤 5：产出修复报告

按 SKILL.md 中的报告模板生成 `fix-report.md`，路径遵循：
1. 查 `docs/` 有无同名功能目录 → 产出到该目录作为补充
2. 无 → 询问用户是否创建独立目录，拒绝则产到项目根目录
