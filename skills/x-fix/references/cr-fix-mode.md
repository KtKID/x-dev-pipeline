# 模式 2：CR 报告修复

> 当用户提供了 CR 报告（`reports/cr/cr-report-*.md`）或明确提到"按 CR 报告修复"时执行。
> 严格按以下步骤执行，不可跳步。

## 步骤 1：定位报告

报告查找优先级：
1. 用户在指令中显式指定路径 → 直接使用
2. 当前仓库 `reports/cr/` 下最新的 `cr-report-*.md`
3. 兼容迁移期的当前工作目录或 task 目录下最新的 `cr-report-*.md`

**无报告时**：立即停止，输出：
```
❌ 未找到 CR 报告，请先运行 x-cr 生成报告。
```

**报告格式异常时**：立即停止，输出：
```
❌ 报告结构不符合预期，无法解析（期望包含"审查结论"表格和"问题详情"章节）。
```

---

## 步骤 2：解析问题清单

1. 扫描「审查结论」各 P 级表格，收集状态为 ❌ 或 ⚠️ 的行
2. 提取每行：严重程度（所属 P 级）、文件:行号、问题描述
3. 扫描「问题详情」章节，提取对应 Bn 编号的完整信息
4. 将审查结论表格条目与问题详情 Bn 条目关联

**已含 `✅已修复` / `➖无需修复` / `⏭已跳过` 标记的条目**：跳过（已被之前的 x-fix 处理过）。

---

## 步骤 3：确定处理范围

- **P0 / P1 / P1.5**：必须处理
- **P2**：处理
- **P3**：默认**跳过**

---

## 步骤 4：汇报计划

修复前向用户汇报：

```
📋 修复计划

将处理：
- B1 (P0) xxx.ts:32 — 条件判断导致错误分支执行
- B2 (P1) xxx.ts:80 — Promise 错误未被捕获

将跳过（P3）：
- B4 (P3) zzz.ts:90 — 建议使用更合理的抽象

共 N 条需处理，M 条跳过。
```

如用户希望调整范围，先按用户指示调整，再继续。

---

## 步骤 5：逐条处理（串行）

### a) 读取上下文代码

读取问题所在文件，定位到报告中标注的行号，读取前后 10~20 行上下文。

**行号偏移处理**：
- 行号处内容与报告描述不匹配 → 通过代码特征（函数名、关键词）重新定位
- 重新定位失败 → 暂停该条，询问用户

**文件不存在**：记录 `➖无需修复`，备注"文件不存在，可能已重构或删除"，继续下一条。

### b) 判断处置情形

| 情形 | 含义 | 操作 |
|------|------|------|
| A | 确认是问题，按报告建议修复 | 执行代码修改 |
| B | 确认是问题，有更优方案 | 执行修改，备注中说明原因 |
| C | 误报，实际不需要修复 | 不修改代码，记录 `➖无需修复` |
| D | 影响较大或无法确定，需用户确认 | 暂停，向用户描述情况并征询意见 |

**多条问题互相影响**：修复 Bn 时已连带解决 Bm → 处理 Bm 时记录 `✅已修复`，备注"已被 Bn 修复消除"，不重复修改。

### c) 执行代码修改（情形 A/B）

- 精准 Edit，只改动必要的代码
- 不引入无关格式调整或重构

### d) 更新报告

每条处理完后立即更新报告：

1. 在「审查结论」表格中替换状态标记：
   - ❌ → `✅已修复` 或 `➖无需修复`
   - ⚠️ → `✅已修复` 或 `➖无需修复`

2. 在对应 `### Bn:` 条目末尾追加：

   ```markdown
   **处置结果**：✅已修复 / ➖无需修复 / ⏭已跳过
   **处置说明**：[简述处置原因或修复方案]
   ```

3. 同步在 `reports/fix/fix-report-YYYYMMDD-HHmmss.md` 或 `reports/fix/fix-note-YYYYMMDD-HHmmss.md` 记录本次修复过程

---

## 步骤 6：追加修复备注

在报告文件末尾追加：

```markdown
---
## 修复备注
> 修复执行时间：YYYY-MM-DD HH:mm

| # | 严重程度 | 文件 | 处置结果 | 修复方式 | 备注 |
|---|----------|------|----------|----------|------|
| B1 | P0 | `xxx.ts:32` | ✅已修复 | 同方案 | |
| B2 | P1 | `xxx.ts:80` | ✅已修复 | 改用 try-catch 包裹整个 async 块 | 报告建议范围偏小 |
| B3 | P2 | `yyy.ts:15` | ➖无需修复 | - | 误报：上游已有空值保护 |
| B4 | P3 | `zzz.ts:90` | ⏭已跳过 | - | P3 优化建议，暂缓 |

### 汇总
| 处置类型 | 数量 |
|----------|------|
| ✅ 已修复 | X |
| ➖ 无需修复（误报） | X |
| ⏭ 已跳过（P3） | X |
```

---

## 边界情况

| 情况 | 处置方式 |
|------|----------|
| 无 CR 报告 | 停止，提示先运行 x-cr |
| 报告格式异常 | 停止，提示结构不符合预期 |
| 文件不存在 | 记录 `➖无需修复`，备注"文件不存在" |
| 行号偏移但能重新定位 | 自动定位后继续处理 |
| 行号偏移且无法定位 | 暂停该条，询问用户 |
| 多条问题互相影响 | 修复时一并处理，后续条目记录已被消除 |
| P3 条目 | 默认跳过，记录 `⏭已跳过` |
| 已有修复标记 | 跳过该条，不重复处理 |

---

## 步骤 7：提醒提交

修复完成后，提示用户：

```
✅ 修复完成，可提交：

git add .
git commit -m "fix: [一句话描述修复内容]"
```

如果本次涉及多个不相关改动，建议按实际修改分开 commit。
