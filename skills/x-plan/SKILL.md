# x-plan 计划生成器

## 核心定位

x-plan 是计划生成工具，负责：
- 基于 `README.md` 中的需求报告制定开发落地方案
- 生成 `plan.md` 和 `dev-checklist.md`
- 展示开发清单并确认优先级
- 将执行权交给 x-dev

x-plan 不负责：
- 需求分析（README.md 由 x-req 负责）
- 需求来源记录
- 代码开发执行
- 任务状态更新

---

## 目录结构

```
dev-pipeline/tasks/<task-name>/
├── README.md      # 完整需求报告 + 状态追踪（x-req 负责）
├── plan.md        # 开发计划（x-plan 负责）
├── dev-checklist.md     # 开发任务（x-plan 负责）
├── changelog.md     # 关键变更（x-dev 负责）
└── artifacts/     # 附件目录
```

**task 根目录固定为 `dev-pipeline/tasks/`**

## 文件模板

README.md 由 x-req 负责创建和维护，包含完整需求报告。x-plan 不修改 README.md 的需求正文，但必须保留并继承其中的“相关文档/相关设计文档”链接，并在完成后更新其「当前阶段」状态。

### plan.md

```markdown
# <task-name> - 开发计划

> 来源：需求分析（README.md）

## 开发策略

[描述整体开发策略]

## 模块落地方式

### 模块 1：[模块名称]
**实现方式**：[描述具体实现方式]
**技术选型**：[技术栈或方案]
**风险点**：[潜在风险]

### 模块 2：...

## 分阶段实施顺序

1. **第一阶段**：基础功能（模块 A）
2. **第二阶段**：核心功能（模块 B）
3. **第三阶段**：增强功能（模块 C）

## 依赖处理

- 外部依赖：[列出]
- 内部依赖：[模块间依赖关系]

## 风险点及应对

| 风险 | 应对方案 |
|------|----------|
| 风险 1 | 方案 1 |

---

*计划生成时间：<date>*
```

### dev-checklist.md

```markdown
# <task-name> 开发清单

| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ⏳ 未开始 | <任务标题> | <备注> |
| #2 | P1 | ⏳ 未开始 | <任务标题> | <备注> |
| #3 | P2 | ⏳ 未开始 | <任务标题> | <备注> |

---

*计划生成时间：<date>*
```

## 执行步骤

1. **解析输入**：提取功能名称
2. **规范化名称**：将功能名称转换为 URL 友好的 task 名
   - 空格 → `-`
   - 斜杠 → `-`
   - 特殊字符去除
3. **查找现有 task**：在 `dev-pipeline/tasks/` 下查找
   - 先尝试精确匹配
   - 再尝试模糊匹配
   - 按相似度排序候选结果
4. **检查前置条件**：
   - `README.md` 必须包含完整需求报告（含「功能模块拆分」章节）才能生成计划
   - 如果 `README.md` 不存在或缺少需求内容，提示用户先运行 x-req
5. **处理查找结果**：
   - 找到唯一匹配：检查并补齐缺失文件（plan.md、dev-checklist.md）
   - 找到多个匹配：询问用户确认
   - 未找到：创建新 task 目录
6. **生成计划**：基于 `README.md` 中的需求报告编写 `plan.md` 和 `dev-checklist.md`
   - 读取并保留 README.md 中的“相关文档”与“相关设计文档”信息
   - 如果 README.md 已存在相关链接，优先补齐而不是覆盖
7. **确定开发顺序**：
   - 按照 P0 → P1 → P2 的顺序排列开发任务
   - 同一优先级内，按依赖关系排序（无依赖的模块优先）
8. **展示开发清单**：列出所有任务和优先级
9. **优先级确认**：
   - 确认 P0/P1/P2 分类是否合理
   - 如果对 README.md 中的优先级排列有不同意见，给出建议并询问用户
10. **更新 README.md**：更新「当前阶段」状态为"开发计划"
   - 若 README.md 中已有相关文档链接，必须保留并同步，不要在更新阶段时丢失原始文档入口
11. **结束**：告知用户计划已准备完成，可以运行 `x:dev <task-name>` 开始执行

## 状态规范

| 符号 | 状态 | 说明 |
|------|------|------|
| ⏳ | 未开始 | 任务待处理，等待开始 |
| ▶️ | 进行中 | 任务正在执行中 |
| 🟡 | 待测试 | 开发完成，等待测试 |
| 🔴 | 测试失败 | 测试未通过，需要修复 |
| 🟢 | 测试通过 | 测试通过，等待最终确认 |
| ✅ | 已完成 | 任务已全部完成并确认 |

## 优先级规范

| 优先级 | 说明 |
|--------|------|
| P0 | 阻塞性问题，必须立即修复 |
| P1 | 重要功能，必须完成 |
| P2 | 优化或增强，可后续处理 |

### 开发顺序规则

- **按优先级排序**：P0 → P1 → P2
- **同优先级内**：按依赖关系排序，无依赖的模块优先
- **优先级确认**：如果对 README.md 中的优先级排列有不同意见，可以给出建议并询问用户确认

## 查找策略

```
输入 → 精确匹配 → 模糊匹配 → 相似度排序 → 必要时确认
```

1. **精确匹配**：查找 `dev-pipeline/tasks/<输入>/README.md`
2. **模糊匹配**：使用 Glob 查找 `dev-pipeline/tasks/<输入>*/`
3. **相似度排序**：按名称相似度排序多个候选
4. **歧义处理**：有多个相似 task 时，询问用户确认

## 创建规则

- **未匹配到任何现有 task 时**：
  - 如果 `README.md` 包含完整需求报告，创建 plan.md 和 dev-checklist.md
  - 如果 `README.md` 不存在或缺少需求内容，提示用户先运行 x-req
- **命中现有 task 时**：默认复用现有目录和文件，不重复创建
- **若命中现有 task 但缺少文件**：自动补齐 `plan.md` 或 `dev-checklist.md`

## 注意事项

- x-plan 不负责需求分析（README.md 由 x-req 负责）
- x-plan 不负责代码开发执行
- x-plan 不负责任务状态更新（由 x-dev 负责）
- x-plan 不负责变更记录（changelog.md 由 x-dev 维护）
- 生成的任务编号必须唯一且按顺序
- task 目录固定在 `dev-pipeline/tasks/` 下
- `README.md` 缺少需求内容不进入 x-plan
