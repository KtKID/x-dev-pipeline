# x-dev 执行规则参考

## 状态规范

| 符号 | 状态 | 说明 |
|------|------|------|
| ⏳ | 未开始 | 任务待处理，等待开始 |
| ▶️ | 进行中 | 任务正在执行中 |
| 🟡 | 待测试 | 开发完成，等待测试 |
| 🔴 | 测试失败 | 测试未通过，需要修复 |
| 🟢 | 测试通过 | 测试通过，等待最终确认 |
| ✅ | 已完成 | 任务已全部完成并确认 |

状态流转：`⏳ → ▶️ → 🟡 → 🟢 → ✅`（失败时 `🟡 → 🔴 → 修复 → 🟡`）

重要规则：
1. 测试通过不等于已完成
2. 只有在实现完成、测试通过、相关记录已更新后，才能改为已完成
3. x-dev 最多到 🟢，✅ 由 review 确认后更新

---

## 开发清单解析规则

从 `dev-checklist.md` 中解析：

```markdown
| 编号 | 优先级 | 质检 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|------|
| #1 | P0 | 🔍 | ⏳ 未开始 | <任务标题> | <备注> |
```

解析要求：
1. 读取表格中每一行任务
2. 识别编号、标题、状态、备注
3. 默认优先执行"进行中" > "测试失败" > "未开始"的任务
4. 跳过"已完成"的任务
5. 如果存在依赖关系，优先执行前置任务

---

## 执行规则

### 1. 开始任务

严格按优先级执行（P0 > P1 > P2）。开始时将状态改为 ▶️ 进行中。

### 2. 完成开发

开发完成但尚未测试 → 改为 🟡 待测试，记录实现内容。

### 3. 执行测试

- 通过 → 🟢 测试通过
- 失败 → 🔴 测试失败，记录失败原因，修复后重新测试

### 4. 关闭任务

满足全部条件才能改为 ✅ 已完成：
- 代码实现完成
- 测试通过
- dev-checklist.md、changelog.md 已同步更新
- 没有明显阻塞项

## 开发清单更新步骤

### 更新时机

1. **开始任务时** — ⏳ → ▶️
2. **开发完成时** — ▶️ → 🟡
3. **测试失败时** — 🟡 → 🔴
4. **测试通过时** — 🟡/🔴 → 🟢

### 更新示例

**原始状态：**
```markdown
| #1 | P0 | ⏳ 未开始 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 开始 → 开发完 → 测试失败 → 修复通过：**
```markdown
| #1 | P0 | 🟢 测试通过 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

### 重要规则

1. 只更新当前正在处理的任务——不批量修改
2. 每完成一个阶段立即更新——不等最后一起改
3. 保持符号统一——使用规范定义的符号
4. 同步更新变更记录——每次状态变化同时写 changelog
5. 不标记为已完成——x-dev 只到 🟢

## 变更记录更新规则

以下情况必须更新 `changelog.md`：
- 需求或范围变更
- 关键技术决策变更
- 测试失败及修复结论
- 阻塞问题及解除结果

记录格式：

```markdown
| 时间 | 操作 | 内容 |
|------|------|------|
| YYYY-MM-DD | 开始开发 | 开始处理 #1：实现设备 Token 校验 |
| YYYY-MM-DD | 测试失败 | #1 本地测试失败，原因：过期判断逻辑错误 |
| YYYY-MM-DD | 修复完成 | 已修复 #1 过期判断逻辑并重新测试通过 |
```
