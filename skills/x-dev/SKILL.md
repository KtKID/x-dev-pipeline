# x-dev 开发任务执行

## 核心定位

x-dev 是执行型 skill，只负责基于现有计划目录执行开发任务。

x-dev 负责：
1. 读取现有功能目录
2. 读取 README.md（完整需求报告）、plan.md、dev-checklist.md、changelog.md
3. 根据开发清单执行开发、修复和测试
4. 更新开发清单中的任务状态
5. 更新changelog.md
6. 在必要时向用户汇报进度、风险和阻塞项

x-dev 不负责：
- 创建 plan 目录
- 创建 README.md
- 创建 dev-checklist.md
- 创建 changelog.md
- 生成新的计划结构
- 自动提交 git，除非用户明确要求

---

## 输入格式

```text
x-dev <功能名称>
````

或

```text
x-dev <功能目录路径>
```

输入可以是：

* 功能名称，例如：`x-dev 设备 Token 认证`
* 现有目录路径，例如：`x-dev dev-pipeline/tasks/设备 Token 认证`

---

## 目录要求

x-dev 默认基于以下目录结构工作：

```text
dev-pipeline/tasks/<功能名称>/
├── README.md      # 完整需求报告（x-req 负责）
├── plan.md        # 开发计划（x-plan 负责）
├── dev-checklist.md     # 开发任务（x-plan 负责）
└── changelog.md     # 关键变更（x-dev 负责）
```

处理规则：

1. 优先按用户提供的目录路径读取
2. 如果用户提供的是功能名称，则查找 `dev-pipeline/tasks/<功能名称>/`
3. 必须确认以下文件存在：

   * `README.md`（含完整需求报告）
   * `plan.md`
   * `dev-checklist.md`
   * `changelog.md`
4. 如果目录或文件缺失，停止执行，并明确提示用户先通过 x-req + x-plan 或手工补齐计划文件

---

## 执行流程

1. 解析输入，定位功能目录
2. 读取 `README.md`，理解核心目标、范围、功能模块拆分、模块依赖关系
3. 读取 `plan.md`，了解开发策略和分阶段实施
4. 读取 `dev-checklist.md`，识别任务编号、任务标题、状态、备注
5. 读取 `changelog.md`，了解已有决策、历史修改和已知问题
7. 选择状态不是”已完成”的任务作为执行对象，按 P0→P1→P2 顺序执行
8. 按任务逐项进行开发、修复和测试
9. 根据执行结果更新开发清单状态
10. 在 `changelog.md` 中记录关键修改、失败原因、修复动作和完成情况
11. 向用户汇报当前完成情况、未完成项和阻塞项

## 开发清单状态规范

x-dev 只使用以下状态：

## 状态规范

| 符号 | 状态 | 说明 |
|------|------|------|
| ⏳ | 未开始 | 任务待处理，等待开始 |
| ▶️ | 进行中 | 任务正在执行中 |
| 🟡 | 待测试 | 开发完成，等待测试 |
| 🔴 | 测试失败 | 测试未通过，需要修复 |
| 🟢 | 测试通过 | 测试通过，等待最终确认 |
| ✅ | 已完成 | 任务已全部完成并确认 |

状态含义：

* **未开始**：尚未开始处理
* **进行中**：正在开发或修复
* **待测试**：开发已完成，等待执行测试
* **测试失败**：测试未通过，需要继续修复
* **测试通过**：测试通过，等待最终关闭
* **已完成**：任务已完成，且相关文档已同步更新

重要规则：

1. 测试通过不等于已完成
2. 只有在实现完成、测试通过、相关记录已更新后，才能改为已完成
3. 如果任务执行中发现问题，应先改为进行中或测试失败，而不是直接标记已完成

## 开发清单解析规则

x-dev 需要从 `dev-checklist.md` 中解析以下结构：

```markdown
## 开发清单

| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ⏳ 未开始 | <任务标题> | <备注> |
| #2 | P1 | ⏳ 未开始 | <任务标题> | <备注> |
| #3 | P2 | ⏳ 未开始 | <任务标题> | <备注> |
```

解析要求：

1. 读取“开发清单”表格中的每一行任务
2. 识别任务编号、任务标题、状态、备注
3. 默认优先执行状态为“进行中”“测试失败”“未开始”的任务
4. 跳过状态为“已完成”的任务
5. 如果任务存在依赖关系，优先执行前置任务

---

## 执行规则

### 1. 开始任务

### 严格按照优先级执行
* 从P0优先级最高，P0>P1>P2>P3

当开始处理某个任务时：

* 将任务状态改为“进行中”

### 2. 完成开发

当某个任务开发完成但尚未测试时：

* 将任务状态改为“待测试”
* 记录完成的实现内容

### 3. 执行测试

测试后根据结果更新：

* 如果通过：改为“测试通过”
* 如果失败：改为“测试失败”，并记录失败原因
* 如果测试失败：针对bug进行修复，修复后继续测试，如果测试通过则改为“测试通过”

### 4. 关闭任务

只有在以下条件满足时才能改为“已完成”：

* 代码实现完成
* 测试通过
* README.md、dev-checklist.md、changelog.md 已同步更新（如有必要）
* 没有明显阻塞项

## 开发清单更新步骤

x-dev 在执行任务过程中，必须实时更新 `dev-checklist.md` 中的任务状态。

### 更新时机

1. **开始任务时** - 将状态从 "⏳ 未开始" 改为 "▶️ 进行中"
2. **开发完成时** - 将状态从 "▶️ 进行中" 改为 "🟡 待测试"
3. **测试失败时** - 将状态从 "🟡 待测试" 改为 "🔴 测试失败"
4. **测试通过时** - 将状态从 "🔴 测试失败" 或 "🟡 待测试" 改为 "🟢 测试通过"

**注意：** x-dev 不负责将任务标记为 "✅ 已完成"，该状态由 review 环节确认后更新。

### 更新示例

**原始状态：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ⏳ 未开始 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**开始处理 #1 后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ▶️ 进行中 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 开发完成后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | 🟡 待测试 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 测试失败后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | 🔴 测试失败 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 修复并测试通过后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | 🟢 测试通过 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

### 状态流转图

```
⏳ 未开始 → ▶️ 进行中 → 🟡 待测试 → 🟢 测试通过
                           ↓
                       🔴 测试失败
                           ↓
                       🟡 待测试（修复后）
                           ↓
                       🟢 测试通过
```

**注意：** "✅ 已完成" 状态由 review 环节确认后更新，不属于 x-dev 的职责范围。

### 重要规则

1. **只更新当前正在处理的任务** - 不要批量修改所有任务状态
2. **每完成一个阶段必须立即更新** - 不要等到所有工作完成才更新
3. **保持符号统一** - 必须使用规范中定义的符号，不要混用其他符号
4. **同步更新变更记录** - 每次更新状态时，同时在 `changelog.md` 中记录原因和结果
5. **不标记为已完成** - x-dev 只负责到 "🟢 测试通过"，不将任务标记为 "✅ 已完成"

## 变更记录更新规则

* 需求或范围变更
* 关键技术决策变更
* 测试失败及修复结论
* 阻塞问题及解除结果

都要更新 `changelog.md`。

至少记录以下内容：

* 时间
* 操作类型
* 任务编号
* 修改内容
* 测试结果
* 失败原因或阻塞项（如有）

推荐记录格式：

```markdown
## 计划记录

| 时间 | 操作 | 内容 |
|------|------|------|
| <date> | 开始开发 | 开始处理 #1：实现设备 Token 校验 |
| <date> | 测试失败 | #1 本地测试失败，原因：过期判断逻辑错误 |
| <date> | 修复完成 | 已修复 #1 过期判断逻辑并重新测试通过 |
```

## README.md 使用规则

执行前必须先阅读 `README.md`（完整需求报告），重点关注：

* 核心目标
* 范围（包含/不包含）
* 功能模块拆分和依赖关系
* 澄清问题及结论

执行过程中不得偏离 `README.md` 中定义的范围。
如果发现任务与 README.md 的目标或范围冲突，应先提示用户，而不是擅自扩展需求。

---

## 输出要求

每次执行后，输出内容应尽量包含：

1. 当前处理的任务编号和标题
2. 已完成的修改
3. 当前任务状态变化
4. 测试结果
5. 是否已写入开发清单和变更记录
6. 剩余未完成任务
7. 风险或阻塞项

---

## 异常处理

### 找不到目录

如果找不到目标功能目录：

* 明确提示未找到对应 `dev-pipeline/tasks/<功能名称>/`
* 提示用户先通过 x-req + x-plan 创建 task 目录或提供正确路径

### 缺少计划文件

如果缺少 `README.md`、`plan.md`、`dev-checklist.md` 或 `changelog.md`：

* 不继续执行开发
* 明确指出缺少哪些文件
* 提示用户先通过 x-req + x-plan 补齐计划文件

### 开发清单格式不合法

如果 `dev-checklist.md` 中没有标准表格或缺少状态列：

* 提示开发清单格式不符合要求
* 要求用户先修正文档，再继续执行

---

## 注意事项

* 不负责创建计划，只消费已有计划
* 不得跳过 README.md 直接开发
* 不得忽略开发清单状态直接执行
* 不得将“测试通过”直接视为“已完成”
* 更新状态时必须同步更新变更记录
* 若用户未明确要求，不自动执行 git 提交
