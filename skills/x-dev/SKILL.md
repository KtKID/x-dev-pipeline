# x-dev 开发任务执行

## 核心定位

x-dev 是执行型 skill，只负责基于现有计划目录执行开发任务。

x-dev 负责：
1. 读取现有功能目录
2. 读取 README.md（需求 + 技术设计 + DoD）、dev-checklist.md、changelog.md
3. 根据开发清单执行开发、修复和测试
4. 更新开发清单中的任务状态
5. 更新changelog.md
6. 在必要时向用户汇报进度、风险和阻塞项

x-dev 不负责：
- 创建 plan 目录
- 创建 README.md
- 创建 dev-checklist.md
- 创建 changelog.md
- 生成新的计划结构
- 自动提交 git，除非用户明确要求

---

## 输入格式

```text
x-dev <功能名称>
````

或

```text
x-dev <功能目录路径>
```

输入可以是：

* 功能名称，例如：`x-dev 设备 Token 认证`
* 现有目录路径，例如：`x-dev dev-pipeline/tasks/设备 Token 认证`

---

## 目录要求

x-dev 默认基于以下目录结构工作：

```text
dev-pipeline/tasks/<功能名称>/
├── README.md          # 需求 + 技术设计 + DoD（x-req 负责）
├── dev-checklist.md   # 开发清单（x-req 负责）
├── changelog.md       # 关键变更（x-dev 负责）
└── plan.md            # [可选/历史] 旧 task 可能有，新 task 不再产出
```

处理规则：

1. 优先按用户提供的目录路径读取
2. 如果用户提供的是功能名称，则查找 `dev-pipeline/tasks/<功能名称>/`
3. 必须确认以下文件存在：

   * `README.md`（含需求 + 技术设计 + DoD）
   * `dev-checklist.md`
   * `changelog.md`（不存在则创建空文件）
4. `plan.md` **不再必需**——新 task 不产出 plan.md；旧 task 如有则仍可读取
5. 如果目录或关键文件缺失，停止执行，提示用户先通过 x-req 补齐

---

## 执行流程

1. 解析输入，定位功能目录
2. 读取 `README.md`，理解核心目标、技术设计、DoD
3. **如果 README.md "涉及模块" 段引用了 docs/ 模块文档 → 跟读该模块文档**，获取接口/数据结构/架构上下文
4. 如果存在 `plan.md`（历史 task）→ 读取了解开发策略（新 task 无此文件则跳过）
5. 读取 `dev-checklist.md`，识别任务编号、任务标题、状态、备注
6. 读取 `changelog.md`，了解已有决策、历史修改和已知问题
7. 选择状态不是”已完成”的任务作为执行对象，按 P0→P1→P2 顺序执行
8. **并行判断**：同一优先级内，将无依赖的任务分组为可并行批次（见”并行开发”章节）
9. 可并行任务通过子 agent 同时开发；有依赖的任务串行执行
10. 根据执行结果更新开发清单状态
11. 在 `changelog.md` 中记录关键修改、失败原因、修复动作和完成情况
12. 向用户汇报当前完成情况、未完成项和阻塞项

---

## 并行开发

同一优先级内有 2+ 个无依赖的待处理任务时，用 Agent 工具创建子 agent 并行开发。

**依赖判断**：备注列标注"依赖 #N"、README.md 声明模块依赖、或两任务改同一文件同一区域 → 有依赖，串行。否则可并行。

**子 agent 规则**：
- 在同一条消息中发出所有 Agent 调用（真正并行）
- 每个子 agent prompt 包含：功能目录路径、任务编号/描述、相关需求上下文
- 子 agent **只改自己任务的代码，不更新 dev-checklist.md 和 changelog.md**（主流程统一更新，避免写冲突）
- 某个子 agent 失败不阻塞其他任务，失败任务回到待处理队列

**结果收集**：所有子 agent 返回后，主流程统一更新状态和 changelog，再逐个进入测试 → x-cr 流程。

**质检标记 🔍**：dev-checklist 质检列标记 🔍 的任务，子 agent 完成后必须立即创建质检 agent 按 x-cr 技能审查，审查通过才继续下一个任务。未标记的任务走常规连续开发流程。

**不并行的情况**：仅 1 个待处理任务、所有任务有依赖链、用户指定只处理某项任务（如"x-dev #3"）。

## 开发清单状态规范

x-dev 只使用以下状态：

## 状态规范

| 符号 | 状态 | 说明 |
|------|------|------|
| ⏳ | 未开始 | 任务待处理，等待开始 |
| ▶️ | 进行中 | 任务正在执行中 |
| 🟡 | 待测试 | 开发完成，等待测试 |
| 🔴 | 测试失败 | 测试未通过，需要修复 |
| 🟢 | 测试通过 | 测试通过，等待最终确认 |
| ✅ | 已完成 | 任务已全部完成并确认 |

状态含义：

* **未开始**：尚未开始处理
* **进行中**：正在开发或修复
* **待测试**：开发已完成，等待执行测试
* **测试失败**：测试未通过，需要继续修复
* **测试通过**：测试通过，等待最终关闭
* **已完成**：任务已完成，且相关文档已同步更新

重要规则：

1. 测试通过不等于已完成
2. 只有在实现完成、测试通过、相关记录已更新后，才能改为已完成
3. 如果任务执行中发现问题，应先改为进行中或测试失败，而不是直接标记已完成

## 开发清单解析规则

x-dev 需要从 `dev-checklist.md` 中解析以下结构：

```markdown
## 开发清单

| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ⏳ 未开始 | <任务标题> | <备注> |
| #2 | P1 | ⏳ 未开始 | <任务标题> | <备注> |
| #3 | P2 | ⏳ 未开始 | <任务标题> | <备注> |
```

解析要求：

1. 读取“开发清单”表格中的每一行任务
2. 识别任务编号、任务标题、状态、备注
3. 默认优先执行状态为“进行中”“测试失败”“未开始”的任务
4. 跳过状态为“已完成”的任务
5. 如果任务存在依赖关系，优先执行前置任务

---

## 执行规则

### 1. 开始任务

### 严格按照优先级执行
* 从P0优先级最高，P0>P1>P2>P3

当开始处理某个任务时：

* 将任务状态改为“进行中”

### 2. 完成开发

当某个任务开发完成但尚未测试时：

* 将任务状态改为“待测试”
* 记录完成的实现内容

### 3. 执行测试

测试后根据结果更新：

* 如果通过：改为“测试通过”
* 如果失败：改为“测试失败”，并记录失败原因
* 如果测试失败：针对bug进行修复，修复后继续测试，如果测试通过则改为“测试通过”

### 4. 关闭任务

只有在以下条件满足时才能改为“已完成”：

* 代码实现完成
* 测试通过
* README.md、dev-checklist.md、changelog.md 已同步更新（如有必要）
* 没有明显阻塞项

## 开发清单更新步骤

x-dev 在执行任务过程中，必须实时更新 `dev-checklist.md` 中的任务状态。

### 更新时机

1. **开始任务时** - 将状态从 "⏳ 未开始" 改为 "▶️ 进行中"
2. **开发完成时** - 将状态从 "▶️ 进行中" 改为 "🟡 待测试"
3. **测试失败时** - 将状态从 "🟡 待测试" 改为 "🔴 测试失败"
4. **测试通过时** - 将状态从 "🔴 测试失败" 或 "🟡 待测试" 改为 "🟢 测试通过"

**注意：** x-dev 不负责将任务标记为 "✅ 已完成"，该状态由 review 环节确认后更新。

### 更新示例

**原始状态：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ⏳ 未开始 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**开始处理 #1 后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | ▶️ 进行中 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 开发完成后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | 🟡 待测试 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 测试失败后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | 🔴 测试失败 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

**#1 修复并测试通过后：**
```markdown
| 编号 | 优先级 | 状态 | 任务 | 备注 |
|------|--------|------|------|------|
| #1 | P0 | 🟢 测试通过 | 实现 Token 校验逻辑 | 核心功能 |
| #2 | P1 | ⏳ 未开始 | 添加过期判断 | 依赖 #1 |
```

### 状态流转图

```
⏳ 未开始 → ▶️ 进行中 → 🟡 待测试 → 🟢 测试通过
                           ↓
                       🔴 测试失败
                           ↓
                       🟡 待测试（修复后）
                           ↓
                       🟢 测试通过
```

**注意：** "✅ 已完成" 状态由 review 环节确认后更新，不属于 x-dev 的职责范围。

### 重要规则

1. **只更新当前正在处理的任务** - 不要批量修改所有任务状态
2. **每完成一个阶段必须立即更新** - 不要等到所有工作完成才更新
3. **保持符号统一** - 必须使用规范中定义的符号，不要混用其他符号
4. **同步更新变更记录** - 每次更新状态时，同时在 `changelog.md` 中记录原因和结果
5. **不标记为已完成** - x-dev 只负责到 "🟢 测试通过"，不将任务标记为 "✅ 已完成"

## 变更记录更新规则

* 需求或范围变更
* 关键技术决策变更
* 测试失败及修复结论
* 阻塞问题及解除结果

都要更新 `changelog.md`。

至少记录以下内容：

* 时间
* 操作类型
* 任务编号
* 修改内容
* 测试结果
* 失败原因或阻塞项（如有）

推荐记录格式：

```markdown
## 计划记录

| 时间 | 操作 | 内容 |
|------|------|------|
| <date> | 开始开发 | 开始处理 #1：实现设备 Token 校验 |
| <date> | 测试失败 | #1 本地测试失败，原因：过期判断逻辑错误 |
| <date> | 修复完成 | 已修复 #1 过期判断逻辑并重新测试通过 |
```

## README.md 使用规则

执行前必须先阅读 `README.md`（完整需求报告），重点关注：

* 核心目标
* 范围（包含/不包含）
* 功能模块拆分和依赖关系
* 澄清问题及结论

执行过程中不得偏离 `README.md` 中定义的范围。
如果发现任务与 README.md 的目标或范围冲突，应先提示用户，而不是擅自扩展需求。

## 异常处理

### 找不到目录

如果找不到目标功能目录：

* 明确提示未找到对应 `dev-pipeline/tasks/<功能名称>/`
* 提示用户先通过 x-req 创建 task 目录或提供正确路径

### 缺少计划文件

如果缺少 `README.md` 或 `dev-checklist.md`：

* 不继续执行开发
* 明确指出缺少哪些文件
* 提示用户先通过 x-req 补齐计划文件

### 开发清单格式不合法

如果 `dev-checklist.md` 中没有标准表格或缺少状态列：

* 提示开发清单格式不符合要求
* 要求用户先修正文档，再继续执行

## 自动流转规则（连续开发模式）

> 适用场景：用户以 `x-dev <功能名称>` 触发、并意图一次性连续推进整个 task 时生效。
> 用户明确只想处理某一项任务（如"x-dev #3"、"只处理任务 3"）时，**不启用本节**，按常规执行流程处理完该项后停下。
> AI 根据对话上下文自主判断，不依赖命令行参数。

每完成一个任务（将状态从 `🟡 待测试` 改为 `🟢 测试通过`）后，**必须立即执行以下动作，不等待用户确认、不问”要不要审查”**：

### 1. 写 dev-report.md 后立即调用 x-verify

- 触发方式：先按模板写 `dev-pipeline/tasks/<task>/dev-report.md`（命令清单 + 自检结论），再进入 x-verify 流程
- x-verify 复跑 dev-report.md 中的命令清单做事实验证（不主观判断）
- x-verify 通过后自动衔接 x-qua-gate（Gate ② 质量评审，3 个 opus 子 agent 串行：R1 spec → R2 边界 → R3 测试真实性）
- x-qua-gate 全部 reviewer pass 后，任务才算完成

### 2. 根据质检结果继续

- **x-verify pass + x-qua-gate 全 pass** → 任务状态改为 `✅ 已完成` → 回到步骤”取下一个任务”
- **任一节点 fail** → 进入 x-fix 修复循环，按 4 条回流规则回到对应节点重审
  - 循环通过 → 任务改为 `✅ 已完成` → 取下一个任务
  - 循环因为 fix-attempts 6 次上限停下 → 停止自动流转，输出汇总上报用户

### 3. 取下一个任务

- 按 P0 → P1 → P2 优先级选择 `dev-checklist.md` 中状态为 `⏳ 未开始` 的下一个任务
- 如果还有未完成任务 → 返回执行流程第 1 步，开始新任务
- 如果所有任务都已完成 → 输出最终汇报

### 4. 停止自动流转的条件

只有以下情况才停下等待用户：

| 停止条件 | 动作 |
|---------|------|
| 所有 dev-checklist 任务均为 `✅ 已完成` | 输出最终汇报（见下） |
| verify / R1 / R2 / R3 ↔ x-fix 同一任务累计修复 ≥ 6 次仍有 P0/P1 | 停止自动流转，列出剩余问题 |
| 任务描述与 README.md 需求冲突 | 停下确认需求，不擅自修改需求 |
| x-dev 执行时遇到环境异常、依赖缺失等 BLOCKED 情况 | 停下上报，不重试 |

### 5. 最终汇报格式

所有任务完成或因停止条件退出时，输出：

```markdown
## 连续开发汇报

- 本次 task：<task 名>
- 完成任务数：N / M
- 跳过的 P1.5/P2/P3 问题：X 条（详见 changelog.md）
- 修改的文件：<列出>
- 未能自动完成的任务（如有）：
  - 任务 #Y：累计修复 K 次后仍有 P0/P1 问题，需手动介入
    - 剩余问题：...
```

### 关键约束

- **不等用户确认**：完成任务后不要问”要继续吗”，直接进入 x-verify（→ x-qua-gate）
- **不跳过质检**：即使 x-dev 自信”这次改动很简单不会有问题”，也必须先调用 x-verify
- **不在连续模式下自动 commit**：git 操作仍由用户手动触发
- **不吞掉错误**：BLOCKED 或停止条件触发时，必须完整汇报，不得”尽力而为”跳过

## 任务完成时必须输出 dev-report.md

每个任务完成后，**除 changelog.md 之外**，必须额外在 `dev-pipeline/tasks/<task>/` 下写入 `dev-report.md`，模板见 `skills/x-dev/templates/dev-report-template.md`。

dev-report.md 是 x-verify 的唯一输入。规则：
1. 验证命令清单**至少包含一条测试类命令**；项目无测试框架时必须显式声明 `no-test-framework: true` + 理由。
2. 命令必须可在项目根目录直接运行，**不允许依赖 dev 临时设置的环境变量**。
3. 自检结论段必须由 x-dev 自己运行命令后填写，不能空着或写"应该能跑"。
4. 写完 dev-report.md 后，下游自动衔接 x-verify（不再走老 x-cr）。
