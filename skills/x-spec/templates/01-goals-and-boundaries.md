# 目标与完成标准

## 系统目标

[系统为什么存在——解决什么问题、为谁创造价值]

## 完成目标（DoD）

[系统做完是什么样——可验证的条件，不写空话]

- [ ] 条件 1（如：用户能通过 CLI 看到 token 用量统计）
- [ ] 条件 2（如：数据延迟 < 5 秒）
- [ ] 条件 N

## 范围边界

**包含**：
- ...

**不包含**：
- ...

## 关键约束

- [技术约束：如必须离线可用、必须跨平台]
- [业务约束：如必须兼容旧数据格式]
- [资源约束：如单人开发、2 周内]

## DoD ↔ 模块追溯矩阵

| DoD 条目 | 需要哪些模块 | 需要哪些 task | 状态 |
|---------|------------|-------------|------|
| 条件 1 | 模块 A + 模块 B | task-1 | 待实现 |
| 条件 2 | 模块 C | task-2 | 待实现 |

> 两个方向都要对得上：DoD 条目没有模块支撑 = 漏了实现；模块没有 DoD 对应 = scope creep。
