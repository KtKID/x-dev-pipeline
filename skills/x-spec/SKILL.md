# x-spec 系统方案规划器

## 核心定位

x-spec 是方案型 skill，负责把一个模糊的系统想法整理成可导航、可拆分、可演进的方案文档体系。

x-spec 负责：
1. 理解系统目标、价值、约束和边界
2. 判断该需求是否应该拆系统、拆模块、拆 task
3. 产出系统级方案目录和总导航 README
4. 拆分系统模块，定义模块职责、边界、依赖、状态
5. 标记哪些模块已足够稳定，可以进入 x-req
6. 建立模块文档与 task 的映射关系

x-spec 不负责：
- 生成开发清单
- 进入代码实现
- 直接修改业务代码
- 替代 x-req / x-plan / x-dev

**核心边界：**
x-spec 回答“这个系统整体应该怎么组织”，不回答“现在具体开发哪一项代码”。

---

## 适用场景

以下情况优先使用 x-spec：
- 用户给的是模糊想法，而不是明确功能需求
- 涉及多个模块、多个阶段、多个 task
- 方案讨论比代码实现更重要
- 当前不确定该不该拆分
- 一个文档可能会非常长，需要拆成导航 + 子文档
- 需要先形成系统级共识，再进入开发

以下情况不需要 x-spec：
- 小功能、小修复、单文件改动
- 已经有稳定需求报告（README.md），只差 plan/dev
- 用户明确要求立即开发一个清晰范围的功能

## 文档生成规则

x-spec 默认只生成以下 3 个文件：
- `README.md`：系统总导航、模块状态、下一步建议
- `02-模块拆分.md`：模块职责、边界、依赖、状态
- `90-task-map.md`：模块到 task 的映射关系

以下文件按需生成，不强制：
- `01-目标与边界.md`：当目标、边界、不包含项较多，README 放不下时再拆出
- `03-核心流程.md`：当系统存在明显流程链路、阶段推进、时序关系时生成
- `04-数据与状态.md`：当核心实体、共享数据、状态流转复杂时生成
- `05-验证与演进.md`：当已经需要规划测试策略、灰度路径、演进路线时生成

原则：
- 优先轻量，不为了“完整”而强行生成空文档
- README 必须能独立承担导航职责
- 子文档只在信息密度明显上升时创建

## 工作流程

### 1. 理解系统意图

从用户描述中提炼：
- 系统目标
- 核心问题
- 用户价值
- 关键约束
- 暂不确定点

### 2. 判断是否需要拆系统

满足任意两项，则建议拆模块：
- 涉及多个代码包或层
- 可独立测试的子能力超过 3 个
- 预期会拆成多个 task
- 单文档预计过长
- 模块之间依赖关系明显
- 用户自己也不确定边界

如果不满足，则建议直接进入：
- `x-qdev`：小改动
- `x-req`：单功能但需要澄清与拆解

### 3. 拆分系统模块

每个模块至少定义：
- 模块名称
- 职责
- 包含范围
- 不包含范围
- 依赖
- 风险或讨论点
- 当前状态

模块状态只使用：
- 探索中
- 方案确认
- 可进入 x-req
- 开发中
- 已完成

### 4. 生成导航 README

README 必须包含：
- 系统目标
- 范围边界
- 模块导航
- 模块状态表
- 文档导航
- 当前建议

### 5. 生成 task-map

对每个模块判断：
- 是否需要独立 task
- 建议 task 名
- 前置依赖
- 是否已具备进入 x-req 的条件

### 6. 给出下一步建议

必须给出明确结论：
- 哪些模块继续留在方案层
- 哪些模块可以进入 x-req
- 是否建议现在就拆 task

## README.md 模板

```markdown
# <系统名称>

> 状态：探索中 | 方案确认中 | 可拆 task | 开发中
> 目标：<一句话说明这个系统要解决什么问题>

## 系统目标

[描述系统存在的原因与价值]

## 范围边界

**包含**：
- ...

**不包含**：
- ...

## 模块导航

| 模块 | 状态 | 文档 | 是否可进 task | 说明 |
|------|------|------|---------------|------|
| 模块 A | 探索中 | `./02-模块A.md` | 否 | 还有边界未定 |
| 模块 B | 可进入 x-req | `./03-模块B.md` | 是 | 可单独开发 |

## 文档导航

- [模块拆分](./02-模块拆分.md)
- [Task 映射](./90-task-map.md)
- [目标与边界](./01-目标与边界.md)（如有）
- [核心流程](./03-核心流程.md)（如有）
- [数据与状态](./04-数据与状态.md)（如有）
- [验证与演进](./05-验证与演进.md)（如有）

## 当前建议

1. 先补充模块 A 的边界
2. 模块 B 可进入 x-req
3. 模块 C 暂不拆 task
```

## 输出要求

完成后，必须向用户明确说明：
1. 该系统是否应该拆分
2. 推荐拆成几个模块
3. 哪些模块适合先进入 x-req
4. 哪些模块仍应停留在方案层
5. 方案文档保存路径

如果用户仍然很模糊，优先先产出最小 3 文件，不要过早扩展全部文档。
