# x-req 需求管理工具

两步确认的需求管理流程，将需求拆分为可执行的功能模块，输出需求分析结果。

## 核心定位

x-req 是需求分析型 skill，只负责需求理解、澄清和模块拆分。

x-req 负责：
1. 理解需求，提炼目标、范围、价值、约束
2. 识别不确定点并澄清
3. 拆分成功能模块
4. 给出模块边界、依赖、优先级、建议开发顺序
5. 输出需求分析结果到 `req.md`
6. 更新 `README.md` 摘要

x-req 不负责：
- 创建开发计划目录（plan/）
- 生成开发任务清单（开发清单.md）
- 生成变更记录（变更记录.md）
- 决定具体开发任务状态
- 进入开发执行

**核心边界：** x-req 回答"做什么、为什么做、拆成哪几块"，不回答"具体怎么落成开发任务"。

## 目录结构

* 没有目录则创建对应目录
```
.claude/tasks/<task-name>/
├── README.md      # 总入口（只放摘要）
├── req.md         # 需求分析结果
├── plan.md        # 开发计划（x-plan 负责）
├── 开发清单.md     # 开发任务（x-plan 负责）
├── 变更记录.md     # 关键变更（x-dev 负责）
└── artifacts/     # 附件目录
```

**task 根目录固定为 `.claude/tasks/`**

## 工作流程

### 1. 获取需求

**从对话获取**：
- 用户直接描述需求："我想要一个用户登录功能"
- 提取关键信息

**从文档获取**：
- 用户提供路径："帮我处理 docs/feature-req.md"
- 使用 Read 工具读取文档内容

---

### 2. 第一步确认：理解需求并解释

**分析要点**：
- 核心目标：这个需求要解决什么问题？
- 主要功能：包含哪些核心功能点？
- 用户价值：为谁创造什么价值？
- 范围边界：哪些做、哪些不做？
- 不确定点：哪些地方需要澄清？

**输出格式**：
```markdown
## 需求理解

**核心目标**：[一句话概括]

**主要功能**：
- 功能点 1
- 功能点 2
- ...

**用户价值**：[描述]

**范围边界**：
- 包含：[列出包含的范围]
- 不包含：[列出不包含的范围]

**需要澄清的问题**：
- 问题 1
- 问题 2
```

**暂停等待用户确认**：使用 AskUserQuestion 询问"理解是否正确？"

---

### 3. 第二步确认：功能模块拆分

拆分原则：
- 每个模块独立可开发、可测试
- 模块间低耦合、可并行执行
- 每个模块有明确边界和价值
- 模块粒度适中，避免过大或过小

**拆分格式**：
```markdown
## 功能模块拆分

### 模块 1：[模块名称]
**职责**：[一句话描述]
**功能范围**：
- 功能点 A
- 功能点 B
**依赖**：[依赖的其他模块或外部系统]
**价值**：[用户价值或技术价值]
**建议优先级**：P0/P1/P2

### 模块 2：...

## 模块依赖关系

[依赖关系图或描述]

## 推荐开发顺序

1. **第一步**：模块 A（原因：无依赖，核心价值）
2. **第二步**：模块 B（原因：依赖模块 A，扩展价值）
3. **第三步**：模块 C（原因：增强功能）

## 澄清问题及结论

| 问题 | 结论 |
|------|------|
| 问题 1 | 用户回答 1 |
| 问题 2 | 用户回答 2 |
```

**暂停等待用户确认**：使用 AskUserQuestion 询问"模块拆分是否合理？"

---

### 4. 保存需求分析结果

用户确认后，保存到 task 目录的 `req.md`：

```
.claude/tasks/<task-name>/
└── req.md    # 需求分析结果
```

#### 更新 README.md 摘要

保存 req.md 后，更新 `.claude/tasks/<task-name>/README.md` 中的摘要：

```markdown
# [需求名称]

> 状态：需求分析中 | 开发中 | 已完成
> 当前阶段：需求分析 → 开发计划 → 开发执行 → 测试 → 关闭

## 核心目标

[一句话概括]

## 模块依赖摘要

- 模块 A → 模块 B → 模块 C
- 核心路径：无依赖模块优先

## 推荐开发顺序

1. 模块 A（无依赖，P0）
2. 模块 B（依赖 A，P1）
3. 模块 C（独立，P1）

## 文件导航

- [需求分析](./req.md)
- [开发计划](./plan.md)
- [开发清单](./开发清单.md)
- [变更记录](./变更记录.md)
```

---

### 5. 完成确认

保存后，向用户报告：
- 创建的 task 目录路径（`.claude/tasks/<task-name>/`）
- 包含的文件列表（README.md、req.md）
- 推荐下一步：选择同一 task 进入 x-plan

---

## 目录命名规范

- **task 目录**：`.claude/tasks/<task-name>/`
- **task 名称**：使用 URL 友好格式（空格转为 `-`，斜杠转为 `-`，特殊字符去除）
- **示例**：`用户登录` → `.claude/tasks/user-login/`，`支付/退款功能` → `.claude/tasks/payment-refund/`

**注意**：task 目录固定在 `.claude/tasks/` 下，不使用 `requirements/`、`plan/` 等平级目录。

## 职责边界

### x-req 负责
- 理解需求、澄清和模块拆分
- 输出需求分析结果
- 保存到 `.claude/tasks/<task-name>/req.md`
- 更新 `README.md` 摘要

### x-req 不负责
- 创建开发计划目录（plan.md）
- 生成开发任务清单（开发清单.md）
- 生成变更记录（变更记录.md）
- 决定具体开发任务状态
- 进入开发执行

---

## 注意事项

- 需求理解不清时，主动询问澄清
- 模块拆分粒度适中，避免过大或过小
- 保存前确保用户两步确认都通过
- 目录命名使用英文小写 + 中划线（如 `user-login`）
- **不要跳过确认步骤直接保存**
- **不要创建开发清单.md**
- **不要进入开发执行**
