# Skill 设计方法论

> 本文是建 / 改 `x-*` skill 的准绳。它从一条朴素直觉出发——"skill 要纯粹、产物要清晰"——但这条直觉里有两个**欠定义的口号**会在执行时退化成坏设计。本文先经对抗性检验把口号修正成可操作的判据，再给出建 skill 时的具体建议。
>
> 配套阅读：`docs/spec/原则性适配处理.md`（思维原则的深讲）。本文是它在"如何设计 skill"这个具体问题上的落地。

---

## 0. 核心直觉（出发点，对的）

1. **skill 要够纯粹**——一个 skill 聚焦，匹配"LLM 一次把注意力放在一件事上"，才不容易跑偏。
2. **任务规范 + 产物要写清楚**——把"什么算优化""什么算合格"讲明白，并给实际例子。

方向没错：臃肿多用途的 skill 触发面模糊、逼模型先猜"我现在是哪个模式"、难测试；产物含糊是 AI 工作的头号失败源。

但这两条都是**口号**，照字面执行会出问题。下面逐条修。

---

## 1. 第一条修正：按"内聚"切，不按"一件事"切

### 口号为什么会坏

"一个 skill 只做一件事"是**自指**的：它要回答"边界画在哪"，可"一件事"本身就由边界定义。它没给任何切分判据，只是把问题换了个说法。照字面执行，结果是**过度切分**。

### 反证（就在本仓库里）

- **x-audit-arch 里塞了 5~6 类检查**（架构一致性、单一事实源、奥卡姆、契约、依赖健康）。按纯粹派读法该拆成 5 个 skill——但 `/x-audit-single-source-of-truth` 和 `/x-audit-layering` 分开是荒谬的。它们被归在一起，靠的是"结构性关注点"这个**内聚**，不是"一件事"。
- **x-qa-gate 是三合一**：内部 dispatch R1/R2/R3。它是"一件事"（把门）还是"三件事"（查 spec / 边界 / 测试）？取决于你站哪一层看。"一件事"在这里直接失效。

→ 真正起作用的判据是**高内聚低耦合**——和"模块怎么分"是同一个老问题。"一件事"是口号，内聚才是标尺。

### 过度纯粹的两笔隐藏成本

1. **耦合税搬家**：skill 越纯，粘合它们的契约越多。本仓库的"skill 间契约"表、`.fix-counter` 三方共享、`dev-report.md` schema——**都是切分的代价**。复杂度不会消失，只会从"skill 内部"搬到"skill 之间的缝"里，而缝里的 bug 最难查。
2. **反而伤 LLM**："匹配 LLM 一次只做一件事"只对了一半。帮 LLM 的不是"skill 只做一件事"，是**决策时手里的上下文聚焦、不自相矛盾**。一个 skill 带模型走多步、每步都清楚，它走得很好；过度切分会逼模型在每次 handoff 重新加载上下文、丢掉上一个 skill 已知的信息、重新推导。**碎片化反而让它失焦**。约束是"工作集聚焦"，不是"skill 数量少"。

### 可操作判据

一个 skill 对应 **一个用户意图 + 一个触发面**。判断要不要拆，问三个问题：

- **触发面会不会打架**？两组功能的触发关键词若总是同时出现 → 内聚，留在一起；若各自独立出现 → 拆。
- **共享不共享同一套上下文 / 产物**？共享 → 留；各读各的代码、各出各的报告 → 拆。
- **拆了之后缝里要立多少契约**？缝越多越该慎拆——耦合税可能比收益大。

## 2. 第二条修正：精确钉死"产物契约"，放手"判断"

### 口号为什么会坏

"把怎么算优化一条条列出来"有个致命副作用：**列成 N 条，模型会把清单当穷举，列完即收工，清单外的真问题被自动忽略**。枚举提升 precision，牺牲 recall，还**压制模型自己的判断力**。

### 反证

- **x-audit-perf 列了 5 类性能检查**。一个落在这 5 类之外的真实性能 bug，模型"打完勾"就走——它以为任务完成了。
- 本仓库用的 skill-creator 文档反复警告这个："少写全大写的 MUST/NEVER""解释 why，让模型超越死规则去泛化"。**裸清单正是它反对的**。
- **过度具体的 spec 本身是漂移源**（违反单一事实源）：prose 写得越死，现实能跟它对不上的地方越多，维护负担越重。

### 关键区分：哪些钉死，哪些留白

| 类型 | 处理 | 例子 |
|------|------|------|
| **产物契约**（不变量） | **精确钉死** | 报告路径、schema、severity 分级、验收条件、输出字段名 |
| **判断**（这次哪些实例重要） | **给原则 + 示例，显式留白** | "什么算优化""什么算架构债"——给规则 + 几个实例 + "非穷举，按规则找同类" |

口诀：**principle + 举例，而非 principle = 举例**。

> ❌ 闭合清单：
> ```
> 优化包括：① N+1 ② 大对象拷贝 ③ 同步阻塞
> ```
> （模型查完这三条就停）
>
> ✅ 原则 + 示例 + 留白：
> ```
> 优化 = 在不改变行为的前提下降低资源消耗（时间/内存/IO）。
> 常见实例：N+1、大对象拷贝、同步阻塞……（非穷举）
> 按上面这条定义，主动寻找清单外的同类问题。
> ```

这正是 `docs/spec/原则性适配处理.md` 的写法——定义 + 例子 + "对你最有用的点"，例子是**示意**不是**边界**。

## 3. 补上：口号没覆盖的两个设计轴

上面两条都在讲"skill 内部长什么样"，但决定 skill 成败的还有两个轴：

### 3.1 触发边界是一等设计对象

一个又纯又规范、却永远不触发（或乱触发）的 skill 等于废的。`description` 的关键词决定它何时被自动唤起，是**第一设计对象**。

> 反讽佐证：本仓库一整轮讨论都在吵触发边界——x-cr vs x-audit-arch vs x-audit-style 谁管什么。这恰恰证明边界是头等问题。

**建议**：每个 skill 显式写"不在范围 / 归谁"段（本仓库的 audit skill 已有此段，保持）。相邻 skill 之间，在各自 SKILL.md 里互相点名边界（如 x-audit-arch 顶部那张"与相邻 skill 的边界"表）。

### 3.2 缝即系统

如果 skill 都很纯，那**skill 之间的 handoff 契约就是系统本身**——管线的正确性活在缝里，不在任何单个 skill 里。

**建议**：把交接契约当成一类一等产物，单独立项、单一事实源维护。本仓库的做法值得延续：CLAUDE.md 里一张"skill 间契约"表集中维护所有缝（`dev-report.md` schema、`.fix-counter` 共享规则、reviewer dispatch 模板），任何一条改动先改这张表。

## 4. 修正后的方法论（四条，扛得住对抗）

> 1. **按内聚切，不按"一件事"切**：一个 skill = 一个用户意图 + 一个触发面；判据是高内聚低耦合，不是数数。
> 2. **钉死产物契约，放手判断**：路径 / schema / severity / 验收条件写死；"怎么算 X"给原则 + 示例 + 显式留白，别让清单替模型思考。
> 3. **触发边界是设计对象**：每个 skill 显式写"不在范围 / 归谁"，相邻 skill 互相点名边界。
> 4. **缝即系统**：skill 之间的 handoff 契约单独立项、集中维护、单一事实源。

## 5. 建 / 改 skill 时的实操建议（checklist）

建一个新 skill 前，按顺序过一遍：

**A. 定位**
- [ ] 这个 skill 对应**哪一个用户意图**？一句话说清。说不清 → 还没想透，别急着建。
- [ ] 它和现有哪个 skill 最近？**画出边界**：我管什么、不管什么、重叠处归谁。
- [ ] 触发关键词和邻居会不会撞？撞了 → 要么合并，要么把边界写进双方 description。

**B. 产物**
- [ ] 输出落在哪个**固定路径**？schema / 模板长什么样？（先写模板，再写正文。）
- [ ] severity / 验收标准是否可判定？避免"看情况""酌情"这类无法验收的词。
- [ ] 产物里哪些字段是**给下游 skill 消费的契约**？标出来，进"skill 间契约"表。

**C. 规则正文**
- [ ] 每条规则有没有给 **why**？（解释原理 > 命令式 MUST。）
- [ ] 枚举类规则有没有配**上位原则 + "非穷举"留白**？防止闭合清单。
- [ ] 有没有全大写 MUST/NEVER 满天飞？有 → 黄灯，改成"解释为什么重要"。

**D. 缝**
- [ ] 它从谁那里接输入、给谁交输出？handoff 契约写清没有？
- [ ] 改动有没有动到共享状态（如 `.fix-counter`）？动了 → 先改 CLAUDE.md 契约表。

**E. 验证**
- [ ] 触发：造几条该触发 / 不该触发的真实 query，确认 description 区分得开。
- [ ] 产物：拿一个真实输入跑一遍，看产出符不符合模板和验收标准。

**F. 收尾（本仓库特有）**
- [ ] 新 skill 是否在 CLAUDE.md / README / README_zh 等**所有列举处**登记？（维持文档单一事实源，别让文档自己漂移。）
- [ ] 是否同步到 Codex？（`./install-codex.ps1`）

---

## 6. 一句话

你的**直觉对**（聚焦、产物清晰是 AI 开发的命门），但两个口号都欠定义：

- "一件事" → 换成"**内聚 + 单一触发意图**"
- "一条条列" → 换成"**原则 + 示例 + 显式留白**"

再补上你漏掉的**触发边界**和**缝契约**，才是完整的一套。
