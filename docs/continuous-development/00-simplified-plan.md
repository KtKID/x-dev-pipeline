# 持续开发驱动机制（简化方案 v2）

> 状态：方案确认
> 日期：2026-04-09
> 关系：本文件为**当前主方案**，`README.md` / `02-module-breakdown.md` / `90-task-map.md` 中描述的 v1 方案作为历史参考保留，不再实施。

---

## 方案目标

在 x-dev-pipeline 中建立一套 **"开发 → 审查 → 修复严重 bug → 复审 → 继续"** 的自动循环，让单次 task 内的多个 dev-checklist 任务能连续推进，不在每个 skill 结束时等待用户确认。

核心原则：**只把必须自动化的环节自动化，其余保持原状。**

---

## 为什么不做 v1 完整版

v1 方案对齐 superpowers 的五阶段流水线，但大部分机制对当前项目属于过度设计：

| v1 引入的机制 | 简化方案为何不做 |
|--------------|-----------------|
| x-drive 连续编排层 | 直接在现有 skill 尾部写"自动进入下一步"即可，不需要独立编排器 |
| x-dev-ext subagent 驱动模式 | 主对话顺序执行就够，不需要 subagent 隔离和四状态机 |
| x-finish 四选一收尾 | 不做 PR / 分支自动化，做完就停下汇报 |
| x-req-ext 设计文档输出 | 现有 x-req + x-spec 组合已够用 |
| session-start hook 断点恢复 | 不是当前瓶颈，依赖平台能力 |
| 每小时 checkpoint | 主对话没有真正计时器，简化为"任务结束即汇报" |
| 模型选择策略 | 当前未到需要分级派发的规模 |

**砍掉所有新增 skill**。只改 3 个已有 skill 的尾部规则。

---

## 保留的 v1 核心思想

以下三条是 superpowers 元规则中真正有效的部分，必须保留：

1. **无人类等待的硬编码循环**
   reviewer 发现问题 → 修复 → 复审 → 再有问题再修，全程不停下问用户。

2. **reviewer 不信任 implementer 报告**
   x-cr 必须独立检查 x-dev 改动，不能因为 x-dev 说"完成了"就跳过审查。

3. **skill 文本即程序**
   自动流转规则写在 SKILL.md 尾部，作为硬约束，AI 读到就必须执行，没有"要不要继续"的选择空间。

---

## 核心流程

```
x-dev 取 dev-checklist 下一个未完成任务
   ↓
实现 + 测试 → 状态改为 🟢 测试通过
   ↓（自动，不问用户）
x-cr 审查本次改动（对照 README.md 需求 + 代码质量）
   ↓
审查结果中有 P0 或 P1 问题？
├─ 有 → x-fix 修复 P0/P1 问题
│        ↓（自动）
│       x-cr 复审
│        ↓
│       仍有 P0/P1？
│        ├─ 有 → 继续 x-fix（累计第 N 次）
│        │       └─ N ≥ 6 → 停下上报用户（防死循环）
│        └─ 无 → 任务状态改为 ✅ 已完成
└─ 无 → 记录 P2/P3（不修）→ 任务状态改为 ✅ 已完成
   ↓（自动）
dev-checklist 还有未完成任务？
├─ 是 → 回到 x-dev 取下一个任务
└─ 否 → 停下，输出本次 task 汇总
```

---

## 严重 bug 定义（与 x-cr 现有分级一致）

| 等级 | 含义 | 是否自动修 |
|------|------|-----------|
| **P0** | 功能正确性 / 致命缺陷（调用链错、条件判断错、核心流程中断） | **自动 x-fix** |
| **P1** | 稳定性 / 安全性 / 数据风险 / 资源泄漏 / 并发竞态 | **自动 x-fix** |
| P1.5 | 架构设计 / SOLID 违反 | 记录不自动修（避免循环中做架构级改动） |
| P2 | 可维护性 / 可读性 / 测试质量 | 记录不修 |
| P3 | 优化建议 / 边界增强 | 记录不修 |

**规则**：只有 P0 + P1 触发自动修复循环。P1.5 因为往往涉及重构，放入记录但不进循环，避免 x-fix 在架构问题上反复试探。

---

## 循环停止条件

只有以下四种情况 AI 才停下等用户：

1. **所有 dev-checklist 任务都 ✅ 已完成** → 汇报总结
2. **x-fix 在同一任务上累计执行 ≥ 6 次，仍有 P0/P1** → 停下上报，列出剩余问题
3. **x-dev 发现任务本身与 README.md 冲突** → 停下确认需求
4. **x-dev 执行时报 BLOCKED**（缺依赖、环境异常等） → 停下上报

### 改动 1：skills/x-dev/SKILL.md

在"执行流程"章节最后追加一节：

```markdown
## 自动流转规则（连续开发模式）

每完成一个任务（状态改为 🟢 测试通过）后，**必须立即执行以下动作，不等待用户确认：**

1. 调用 x-cr 审查本次改动（传入本次改动涉及的文件）
2. 根据 x-cr 结果：
   - 无 P0/P1 → 将任务状态改为 ✅ 已完成，取下一个未完成任务
   - 有 P0/P1 → 进入 x-cr ↔ x-fix 修复循环（见 references/auto-loop.md）
3. 所有任务完成后，输出本次 task 汇总
4. 满足"循环停止条件"之一时，停下上报用户
```

### 改动 2：skills/x-cr/SKILL.md

在"报告保存"章节之后追加一节：

```markdown
## 自动流转规则（连续开发模式）

审查完成并保存报告后，**必须立即执行以下动作，不等待用户确认：**

1. 统计 P0 + P1 问题数量
2. 若 P0 + P1 > 0：
   - 立即调用 x-fix，传入本次 CR 报告路径
   - x-fix 修复后，**必须再次调用 x-cr 复审同一批文件**
   - 复审仍有 P0/P1 → 继续 x-fix（累计次数 +1）
   - 累计 ≥ 6 次仍未通过 → 停下上报用户
3. 若 P0 + P1 = 0：
   - 记录 P1.5/P2/P3 到 changelog.md 但不阻塞
   - 将当前任务状态改为 ✅ 已完成
   - 通知 x-dev 取下一个任务

**硬约束：不信任 x-dev 的自我报告。必须独立读取代码对照 README.md 中的需求。**
```

### 改动 3：skills/x-fix/SKILL.md

在 CR 修复模式章节后追加一节：

```markdown
## 自动流转规则（连续开发模式）

修复完成后，**必须立即执行以下动作，不等待用户确认：**

1. 仅修复 P0 和 P1 级问题，跳过 P1.5/P2/P3
2. 修复完成后，**必须调用 x-cr 对同一批文件复审**
3. 复审结果由 x-cr 决定下一步（继续修复 / 进入下一任务 / 停下）

**硬约束：不允许在一次 x-fix 中跨越多个原始任务。一次循环只处理当前任务的问题。**
```

### 新增：各 skill 的 references/ 共享规则

实施时选择了"各 skill 的 references/" 路径，具体落地为 `skills/x-cr/references/auto-loop-mode.md`（x-cr 在循环中充当裁判，规则文件放在 x-cr 下最合理）。

```markdown
# 连续开发自动流转规则

## 循环状态：fix_attempts 的存储位置

fix_attempts 必须在 **文件中** 持久化，而**不是**内存变量。

**原因**：Claude Code 的主对话模型里没有原生的"内存变量"概念。盘点可用机制：

| 机制 | 是否可行 | 说明 |
|------|---------|------|
| 对话上下文自觉记住 | 脆弱 | 上下文压缩会摘要掉数字；跨 session 完全失效；中间插话易遗漏 |
| TodoWrite 工具 | 不支持 | 只能存预定义的几种状态，不能存自定义数值 |
| Auto memory / MEMORY.md | 不合适 | 为长期用户记忆设计，每次循环写入会污染记忆索引 |
| subagent 状态 | 不适用 | 简化方案已放弃 subagent 路线 |
| **项目内文件持久化** | **✅ 推荐** | 与 task 状态存储位置天然一致 |

**权威存储位置**：`dev-checklist.md` 对应任务行的"备注"列，格式 `[fix:N]`：

- 首次进入修复循环时写入 `[fix:1]`
- 每次复审后 +1，更新为 `[fix:2]`、`[fix:3]`……
- 读取时从该列解析 `[fix:N]`，没有则视为 0
- 任务最终 ✅ 时可保留或清理，保留有利于历史追溯

## 循环步骤

```
loop:
  x-cr 审查 → 报告
  if P0+P1 == 0:
    任务 → ✅ 已完成
    break
  读 dev-checklist.md 备注列的 [fix:N]（没有则 N=0）
  if N >= 6:
    停下上报
    break
  x-fix 修复 P0+P1
  写回 [fix:N+1] 到备注列
  goto loop
```

## 停止后汇报格式

```markdown
## 连续开发汇报

- 本次 task：<task 名>
- 完成任务数：N / M
- 跳过的 P2/P3 问题：X 条（详见 changelog.md）
- 未能自动修复的任务：
  - 任务 #Y：累计修复 6 次后仍有 P0/P1 问题，需手动介入
    - 剩余问题：...
```
```

## 实施结果（2026-04-09）

方案于 2026-04-09 通过 `x-qdev continuous-dev-auto-loop` 任务实施，随后又经过一次代码审查发现并修复了一些偏差。以下是与原方案的差异：

### 已实施且符合方案

- ✅ x-dev/SKILL.md 尾部追加"自动流转规则"章节
- ✅ x-fix/SKILL.md 尾部追加"自动流转规则"，严格限定只修 P0/P1
- ✅ P0/P1 自动修，P1.5/P2/P3 记录不修的分级规则
- ✅ 4 种循环停止条件（全完成 / 6 次上限 / 需求冲突 / BLOCKED）
- ✅ x-req / x-plan / x-qdev / x-spec 未做任何改动
- ✅ 不新增 skill

### 与方案的偏差（已授权，已记录）

**偏差 1：x-cr 的规则位置**

- 原方案说"在'报告保存'章节之后追加一节"（留在 SKILL.md 主体）
- 实施时 x-cr 同时经过 `x-cr-optimization` 任务的 skill-creator 规范重构，自动流转规则被下沉到 `skills/x-cr/references/auto-loop-mode.md`
- 功能语义 100% 保留，只是物理位置从 SKILL.md 尾部下沉到独立 reference 文件
- 授权方式：用户明确要求 `/skill-creator` 优化 x-cr 并选 "b 一次性实施"

**偏差 2：`fix_attempts` 从"内存变量"改为"文件持久化"**

- 原方案：L167 "维护一个本任务的内存变量：fix_attempts（初始 0）"
- 实施：`dev-checklist.md` 对应任务行的"备注"列 `[fix:N]`
- 原因：Claude Code 主对话没有原生"内存变量"机制（见前文盘点表）；文件持久化是唯一可靠方案
- 本文件已同步更新"新增共享规则"一节的描述

**偏差 3：x-cr 新增"对照文档检查"流程（超出原方案范围）**

- 原方案的"独立验证硬约束"只在连续开发模式下要求对照 README.md
- 实施后又新增了一层 SKILL.md 主体的通用"对照文档检查"流程（第 0 步定位文档 + 第 3 步对照文档 + 不一致处理），两种模式共用
- 新规则：代码与文档不一致时，先回忆对话上下文和 changelog 的授权痕迹；没找到授权痕迹就停下问用户"改代码 (A) / 改文档 (B) / 误判 (C)"
- 授权方式：用户明确要求"审核流程需要从需求文档开始..."
