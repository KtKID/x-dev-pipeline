核心结论：这些原则在 AI 辅助开发里不是“思维鸡汤”，而是用来把 **AI 生成代码** 变成 **可对齐、可审查、可验证、可回滚的工程流程**。

最重要的链路是：

> 需求和人类理解一致 → 架构方案符合项目规范 → 实现保持最小充分改动 → Review 主动找错 → 自动化测试给证据 → 根据证据更新判断

---

## 1. 第一性原理

**定义：**
第一性原理是把问题拆到最基础、不可再假设的事实和约束，然后从这些基础重新推导方案，而不是直接套经验、套框架、套别人的做法。

它问的是：

> 这个问题真正必须满足什么？哪些只是习惯、假设、历史包袱？

在 AI 辅助开发里的用法：

假设你要做 agent workflow，不应该一开始就问：

> 我要用 LangGraph、AutoGen、CrewAI 还是自己写？

而是先问：

> 一个 agent 完成开发任务，本质需要什么？

可能拆出来是：

1. 用户目标
2. 上下文
3. 可调用工具
4. 状态记录
5. 权限控制
6. 执行步骤
7. 中间证据
8. Review
9. 测试验证
10. 失败恢复

这样推导出来，你就会知道 workflow 不是“多 agent 炫技”，而是为了让复杂任务有 **状态、边界、证据和验证闭环**。

**对你最有用的点：**
第一性原理适合用在产品定位、架构设计、workflow 设计上。它能避免“看到别人用 Rust/多 agent/MCP/插件系统，我也加一个”的盲目扩展。

---

## 2. 贝叶斯公式

**定义：**

贝叶斯公式是：

```text
P(H | E) = P(E | H) * P(H) / P(E)
```

含义是：

> 在看到新证据 E 之后，我们应该如何更新对某个假设 H 的相信程度。

几个变量：

| 符号       | 含义                     |
| -------- | ---------------------- |
| H        | 假设，例如“这个 bug 是配置读取导致的” |
| E        | 证据，例如日志、测试失败、代码路径      |
| P(H)     | 先验概率，看到证据前你觉得它有多可能     |
| P(E | H) | 如果 H 为真，看到这个证据的概率      |
| P(H | E) | 看到证据后，H 成立的概率          |

在 AI 辅助开发里的用法：

AI 很容易“自信地猜”。贝叶斯思维要求它不要只给结论，而要不断根据证据更新判断。

例如：

> 假设：登录失败是 token 过期导致的。
> 证据 1：日志显示 401。这个支持 token 问题。
> 证据 2：刷新 token 后仍失败。这个降低 token 问题概率。
> 证据 3：同一 token 在另一个环境可用。进一步降低 token 问题概率。
> 新假设：环境变量或代理配置问题概率上升。

这比 AI 直接说“应该是 token 过期”可靠得多。

**对你最有用的点：**
贝叶斯思维适合用在 bug 定位、review 判断、测试失败分析、架构风险评估上。

一个好的 agent 不应该说：

> 我认为没问题。

而应该说：

> 当前证据支持 A，但 B 仍未排除。已验证 X、Y，未验证 Z，因此置信度是中等。

## 3. 奥卡姆剃刀

**定义：**
奥卡姆剃刀是指：

> 在多个解释或方案都能解释现象、满足需求时，优先选择更简单、假设更少的那个。

注意，它不是说“永远选最简单的方案”，而是：

> 在满足需求的前提下，不引入不必要的复杂度。

在 AI 辅助开发里的用法：

如果一个需求只是：

> 支持 agent 配置读取。

那么更简单的方案可能是：

```text
TOML 配置文件 → 解析 → 校验 → 构造 AgentConfig
```

而不是一开始就设计：

```text
配置中心 → 动态热更新 → 插件系统 → 分布式配置同步 → 多租户权限模型
```

后者也许未来有用，但当前可能只是过度设计。

**对你最有用的点：**
奥卡姆剃刀适合用在架构设计和代码 review 上，尤其防止 AI 一次性写出“看起来很完整但实际难维护”的大系统。

Review 时可以直接问：

> 这个抽象现在解决了真实问题吗？
> 如果删掉这一层，系统会不会更清楚？
> 有没有更小的改动能满足同样需求？

---

## 4. 对抗性检验

**定义：**
对抗性检验是主动寻找一个方案、结论或代码实现的漏洞，而不是只证明它“看起来对”。

它问的是：

> 在什么情况下这个方案会失败？
> 哪个输入会击穿它？
> 哪个边界条件没有处理？
> 哪个假设一旦不成立，系统就会崩？

在 AI 辅助开发里的用法：

AI 写完代码后，不能只让它总结“我做了什么”，还要让另一个 reviewer 或测试 agent 从反方向攻击：

```text
1. 需求是否被误解？
2. 有没有改错模块？
3. 有没有破坏已有行为？
4. 有没有遗漏异常路径？
5. 有没有并发、权限、状态、缓存问题？
6. 有没有测试只覆盖 happy path？
7. 有没有为了通过测试而写死逻辑？
```

**对你最有用的点：**
对抗性检验是 AI 辅助开发里非常关键的一环，因为 AI 最大的问题不是不会写代码，而是会“自证正确”。

# 5. 还能用到 AI 辅助开发里的重要原则

下面这些原则可以直接变成你的 agent workflow 检查项。

## 原则一：需求一致性原则

**定义：**
代码开始前，必须保证 AI 对需求的理解和人类的真实意图一致。

这是最重要的原则。因为需求错了，后面的架构、代码、review、测试都会变成“高质量地做错事”。

AI 辅助开发里要落地成一个明确产物：

```text
需求理解文档
- 用户真正要解决的问题
- 目标行为
- 非目标行为
- 输入输出
- 约束条件
- 边界情况
- 验收标准
- 示例
- 反例
```

关键检查问题：

> 用户看完这份需求理解，会不会说：“对，我就是这个意思。”

对你的 agent 框架来说，`x-req` 这一层就应该做这个，而不是直接进入开发。

---

## 原则二：规格先行原则

**定义：**
先定义系统应该表现出什么行为，再写实现。

规格不是大文档，而是可验证的行为描述。

例如不要只写：

```text
支持读取 agent 配置。
```

应该写成：

```text
当配置文件存在多个 [[agents]] 时，系统应解析为多个 AgentConfig。
如果 id 重复，应报出明确错误。
如果 model 缺失，应拒绝启动。
如果 desc 为空，可以允许，但需要默认值或警告。
```

这样 AI 才知道怎么实现，review 才知道怎么检查，测试才知道怎么写。

---

## 原则三：证据优先原则

**定义：**
所有结论都尽量基于证据，而不是基于 AI 的语言自信。

在 AI 辅助开发里，证据包括：

```text
- 代码路径
- 函数名
- 日志
- 测试结果
- diff
- benchmark 数据
- 复现步骤
- 失败截图
- 命令输出
```

不合格的 AI 输出是：

```text
我已经修复了问题。
```

合格的 AI 输出是：

```text
修改了 src/config/agent_loader.py 中的 load_agents。
新增 tests/test_agent_loader.py::test_duplicate_agent_id。
执行 pytest tests/test_agent_loader.py 通过。
```

你的 agent 框架如果要做得专业，所有关键结论都应该尽量绑定证据。

---

## 原则四：最小充分改动原则

**定义：**
每次改动应该刚好满足需求，不扩大范围，不顺手重构无关模块。

AI 很容易出现这个问题：

> 修一个 bug，顺便重构半个系统。

这会导致 review 成本暴涨，也更容易引入新问题。

检查问题：

```text
这个 diff 是否只改了必要文件？
有没有无关格式化？
有没有顺手改命名？
有没有改变公共接口？
有没有引入新的抽象层？
```

对 agent workflow 来说，coder agent 应该被约束在：

```text
需求范围内实现。
不做无关重构。
必要时先申请扩大范围。
```

---

## 原则五：架构一致性原则

**定义：**
新代码应该遵循项目已有架构、命名、模块边界和开发规范，而不是 AI 自己发明一套风格。

这对你尤其重要，因为你一直强调模块化、高内聚低耦合、边界类、命名归属感。

AI 辅助开发里要检查：

```text
- 是否放在正确模块？
- 是否复用了已有边界类？
- 是否绕过了已有抽象？
- 是否破坏分层？
- 是否引入循环依赖？
- 命名是否符合项目语义？
- 错误处理方式是否和项目一致？
- 配置、日志、测试风格是否一致？
```

这类检查最好不要靠 AI 自由发挥，而要在项目里沉淀成：

```text
ARCHITECTURE.md
CONVENTIONS.md
REVIEW_CHECKLIST.md
```

然后每次开发前让 AI 读取。

---

## 原则六：契约优先原则

**定义：**
模块之间先定义清楚契约，再写内部实现。

契约包括：

```text
- 输入是什么
- 输出是什么
- 错误怎么返回
- 是否允许为空
- 是否有副作用
- 是否修改状态
- 是否线程安全
- 是否可重复执行
```

例如你做 agent 配置，不只是“读取配置”，而是要定义：

```text
AgentConfigLoader.load(path) -> list[AgentConfig]
```

同时明确：

```text
- 文件不存在：抛 ConfigNotFoundError
- TOML 格式错误：抛 ConfigParseError
- 字段缺失：抛 ConfigValidationError
- id 重复：抛 DuplicateAgentIdError
```

这样代码实现、review 和测试都会更稳。

---

## 原则七：单一事实源原则

**定义：**
同一份关键信息只应该有一个权威来源，避免多个地方重复维护。

AI 很容易复制粘贴配置、枚举、规则、prompt、schema，短期能跑，长期必乱。

在 AI 辅助开发里要检查：

```text
- 同一个字段定义是否重复？
- 同一个默认值是否散落多处？
- 同一个 prompt 规则是否复制了多个版本？
- schema 和文档是否可能不一致？
- 测试是否写死了实现细节？
```

例如 agent 配置字段应该以一个 schema/dataclass/pydantic model 为准，文档和 UI 都从它派生或至少保持同步。

---

## 原则八：可追溯性原则

**定义：**
每个代码改动都应该能追溯到需求、设计、任务、测试和 review 结论。

理想链路是：

```text
需求项 → 设计决策 → 代码 diff → 测试用例 → review 发现 → 修复记录
```

这非常适合你的 x-dev-pipeline / kongming-agent。

如果你要做成 workflow，可以要求每个任务保留：

```text
task_id
requirement_id
design_doc
changed_files
test_commands
review_findings
fix_commits
final_verification
```

这会让 AI 开发从“聊天记录”变成“工程审计记录”。

---

## 原则九：风险分层原则

**定义：**
不同风险的改动，使用不同强度的 review 和测试。

不是所有任务都需要复杂流程。

可以分成：

| 风险   | 示例              | 流程强度                    |
| ---- | --------------- | ----------------------- |
| 低风险  | 文案、注释、小 UI 调整   | 简单 review + smoke test  |
| 中风险  | 新增普通功能          | 需求确认 + 单测 + review      |
| 高风险  | 配置、权限、状态机、数据迁移  | 设计评审 + 对抗 review + 集成测试 |
| 极高风险 | 删除数据、支付、安全、账号系统 | 人工确认 + 回滚方案 + 全链路测试     |

这样你的 workflow 不会过重，也不会对高风险任务放得太松。

---

## 原则十：失败优先原则

**定义：**
设计系统时，优先考虑失败路径，而不是只考虑正常路径。

AI 生成代码最常见的问题是 happy path 很完整，异常路径很薄弱。

检查问题：

```text
文件不存在怎么办？
配置为空怎么办？
字段类型错怎么办？
工具调用超时怎么办？
模型返回非法 JSON 怎么办？
任务执行一半中断怎么办？
用户拒绝权限怎么办？
测试环境和真实环境不一致怎么办？
```

对 agent 框架来说，这个原则非常重要。因为 agent 系统本质是长链路系统，任何一步失败都要能记录、恢复、降级或停止。

---

## 原则十一：自动化验证原则

**定义：**
不要只依赖 AI 解释“为什么代码正确”，而要用自动化测试、静态检查、类型检查、lint、benchmark 给出验证证据。

AI 辅助开发里，最低要求是：

```text
- 修改了逻辑：补单元测试
- 修改了模块交互：补集成测试
- 修改了用户流程：补 e2e 或 smoke test
- 修 bug：先复现，再修复，再加回归测试
- 改性能：给 benchmark 对比
```

特别是 bug fix，最好遵循：

```text
失败测试 → 修复代码 → 测试通过
```

这比“我修好了”可靠得多。

---

## 原则十二：人类最终裁决原则

**定义：**
AI 可以探索、实现、review、测试，但关键语义判断仍然要由人类确认。

尤其是这些问题：

```text
这个需求是不是用户真正想要的？
这个架构方向是不是符合长期规划？
这个复杂度是否值得？
这个 trade-off 是否接受？
这个行为改变是否符合产品意图？
```

AI 可以给建议，但不应该替你决定产品方向和工程边界。

---

# 6. 可以整理成 AI 辅助开发的四道门

你可以把这些原则压成一个实际工作流。

## 第一门：需求一致门

目标：确保 AI 没理解错。

检查项：

```text
- 目标是否明确？
- 非目标是否明确？
- 验收标准是否明确？
- 是否有示例和反例？
- 是否有未解决歧义？
```

输出物：

```text
requirements.md
acceptance_criteria.md
```

核心原则：

```text
第一性原理
需求一致性
规格先行
```

---

## 第二门：架构规范门

目标：确保方案符合项目结构，不乱加复杂度。

检查项：

```text
- 是否遵循现有模块边界？
- 是否符合命名规范？
- 是否保持高内聚低耦合？
- 是否有更小方案？
- 是否引入不必要抽象？
- 是否有回滚方案？
```

输出物：

```text
design.md
implementation_plan.md
risk_assessment.md
```

核心原则：

```text
奥卡姆剃刀
架构一致性
契约优先
最小充分改动
```

---

## 第三门：对抗 Review 门

目标：主动发现问题，而不是让 AI 自证正确。

检查项：

```text
- 需求是否被误解？
- 代码是否改错地方？
- 是否破坏已有行为？
- 是否遗漏异常路径？
- 是否有状态、并发、权限、缓存问题？
- 是否有测试盲区？
- 是否存在过度设计？
```

输出物：

```text
review_report.md
findings.md
fix_plan.md
```

核心原则：

```text
对抗性检验
证据优先
贝叶斯更新
风险分层
```

---

## 第四门：自动化验证门

目标：用机器证据证明当前版本足够可信。

检查项：

```text
- 单元测试是否覆盖核心逻辑？
- 集成测试是否覆盖模块协作？
- e2e/smoke 是否覆盖关键流程？
- bug fix 是否有回归测试？
- 测试命令和结果是否记录？
- 是否有未验证风险？
```

输出物：

```text
test_report.md
verification_log.md
known_risks.md
```

核心原则：

```text
自动化验证
失败优先
可追溯性
人类最终裁决
```

# 7. 最适合你 agent 框架的总结版

你可以把 AI 辅助开发原则压成这句话：

> AI 不应该只是生成代码，而应该围绕需求、架构、实现、review、测试形成一个带证据的闭环。

更工程化一点：

```text
1. 需求必须和人类理解一致。
2. 设计必须从第一性原理推导，并遵循现有架构。
3. 实现必须最小充分，避免无关扩张。
4. Review 必须对抗性地找问题，而不是总结优点。
5. 测试必须自动化，并留下可复现证据。
6. 所有结论都应该根据证据更新，而不是依赖 AI 自信。
```

如果对应到你的 workflow，可以是：

```text
x-req      → 需求一致性
x-design   → 架构规范与最小方案
x-dev      → 受约束实现
x-review   → 对抗性检验
x-test     → 自动化验证
x-fix      → 根据证据修复
x-summary  → 留下可追溯记录
```

这里最核心的不是“多 agent”，而是 **每个阶段都有明确责任、输入、输出、证据和失败处理**。这才是 AI 辅助开发从玩具变成工程系统的分界线。
